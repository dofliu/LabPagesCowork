RAG Benchmark 可重現性危機：17 種檢索模式的多輪穩健性分析

2026-04-27 • 劉瑞弘教授團隊 • RAG_Ultimate / 實驗設計 / 可重現性 / 風電知識庫

RAG Benchmark 可重現性 穩健性分析 實驗設計

「RAPTOR 的 54 次錯誤，是演算法不穩定還是別的原因？」——這個問題讓 RAG_Ultimate 研究團隊在昨日反思了整份 Tier-2 benchmark 報告的詮釋方式。最終答案出乎意料：不是 RAPTOR 出了問題，而是網路斷線（getaddrinfo failed）導致 r03–r05 三輪的多項模式批次失敗。這個教訓深刻地提醒我們：在 AI 研究中，根因分析（root cause analysis） 的嚴謹度，與演算法本身同等重要。

一、RAG Benchmark 的隱性風險：你的實驗結果真的可信嗎？

大型語言模型（LLM）評測的可重現性問題近年已引發廣泛關注。2023 年的著名研究 「Is ChatGPT a Good Recommender?」 指出，相同 prompt 在不同時間點的輸出可能有顯著差異；2024 年 RAGAS 框架的作者也在論文中坦承，faithfulness 指標的多輪跑分標準差可達 ±0.05 以上。對於 RAG pipeline 而言，不確定性來源更為複雜：向量檢索的隨機採樣、LLM 生成的溫度設定、外部 API 的網路延遲，三者疊加，使得「單次跑分即發表」的做法隱藏巨大風險。

DOF Lab 的 RAG_Ultimate 專案正面對這一挑戰。我們構建了涵蓋 4 個風電品牌（Z72、H127、V47、E40）、共 405 道問答題的專業 QA 測試集，並設計了 17 種檢索策略的系統性比較實驗（稱為 Tier-2）。然而，當 5-run aggregate 數據彙整完畢後，我們發現一個令人警醒的現象：12 個可評測模式中，有 9 個因 valid run 數不足（< 4 次），無法達到統計可信標準。

⚠️ 事件重建：RAPTOR 錯誤 54 次的真相

初步報告將 r03–r05 輪次中 RAPTOR 模式的 errors=54 歸因於「演算法遞迴摘要的不穩定性」。然而，經交叉比對 r03/r04/r05 的執行日誌，發現錯誤訊息統一為 socket.gaierror: [Errno 11001] getaddrinfo failed——這是典型的 DNS 解析失敗，與 RAPTOR 演算法本身完全無關。根因確認為：批次測試期間網路連線中斷，導致 LLM API 呼叫全面失敗，錯誤被錯誤地歸咎於演算法層級。

二、穩健性分析框架：如何設計可信賴的 RAG 評測

從這次事件，我們整理出一套 RAG Benchmark 穩健性分析的核心準則。這不只適用於 RAG_Ultimate，也是任何需要多 API 呼叫、大規模自動評測的 AI 研究都應採用的實驗設計原則。

評估面向 不穩健做法 建議做法
實驗輪數 單次跑分（N=1） ≥ 4 完整 run（有效計算信賴區間）
錯誤分類 所有失敗歸因演算法 區分基礎設施失敗 vs. 演算法失敗
中斷處理 失敗就放棄整輪 checkpoint + retry 機制，記錄中斷原因
統計報告 僅報告平均分 報告平均 ± 標準差 + 95% CI
可重現性記錄 僅存最終結果 保留每輪原始 log + valid_record_matrix
異常值偵測 人工事後判斷 自動 outlier flagging（IQR 或 Z-score）

📐 信賴區間目標：為何需要 ≥ 4 次完整 run？

以 RAGAS answer_correctness 指標為例，假設單次標準差 σ ≈ 0.06（實測），要達到 95% 信賴區間寬度 ≤ ±0.03，需要樣本數 n ≥ (1.96 × σ / 0.03)² ≈ 16 次——但這是對「每個 query」而言。對於「每個模式的平均分」，因為我們已有 405 道題，信賴區間寬度壓縮，實際上 4 次完整 run（valid = 104 筆有效評測） 已可達到模式間比較的統計功效。這是 RAG_Ultimate Tier-2 補跑（r06、r07）的目標基準。

三、17 種 RAG 模式的初步排名格局

儘管多輪數據尚未完整，現有的 5-run aggregate 已顯示出幾個清晰的趨勢。在風電設備 QA 場景（以 Z72 品牌為主力測試集）中：

  • Hybrid Search（密集 + 稀疏雙軌）：在有效 run 中一致表現領先，answer_correctness 均值 ≈ 0.71，穩健性最佳（CV < 5%）
  • Contextual Compression RAG：平均分高（≈ 0.69），但跨 run 變異較大，顯示壓縮策略對 prompt 版本敏感
  • Standard RAG（baseline）：穩定但偏低（≈ 0.58），作為比較基準效果良好
  • RAPTOR：在網路正常的 r01、r02 輪次中表現優異（≈ 0.66），但需補跑確認穩定性
  • Self-RAG / Corrective RAG：因依賴多輪 LLM 呼叫，網路斷線影響最嚴重，有效數據最少

這些初步結果與近期 RAG 比較研究的結論一致：混合檢索（hybrid retrieval）在特定領域 QA 上往往優於純密集向量檢索，尤其當文件包含大量技術術語（如風電規格書的部件代號、IEC 標準條款號）時，BM25 的精確匹配能力提供了關鍵補充。

四、對 AI 研究社群的啟示：從「跑出來」到「可信賴」

RAG_Ultimate 的這段經歷折射出當前 AI 研究的一個系統性痛點：評測基礎設施的脆弱性被嚴重低估。當我們把大量精力投入演算法創新時，往往忘記評測本身也需要工程保障。網路、API 配額、記憶體溢位、隨機種子——任何一個環節的失控，都可能讓辛苦跑出的數字變成「美麗的謊言」。

對此，研究社群已有初步回應。Meta 的 PurpleLlama 評測框架、Google 的 BIG-Bench 協議，都強調「evaluation harness」的可靠性與錯誤追蹤。國內方面，學術界開始要求論文附上完整的實驗重現腳本（reproducibility package）。DOF Lab 採用的 checkpoint-resume 機制、valid_record_matrix 追蹤、以及錯誤分類規範，正是在這個方向上的實踐嘗試。

🔬 DOF Lab 研究行動項目

  • RAG_Ultimate Tier-2 補跑：規劃執行 r06、r07，目標每模式達到 ≥ 4 完整 valid run，確保信賴區間 ≤ ±0.03
  • 網路斷線防護機制：在 benchmark 執行腳本中加入 retry with exponential backoff + offline fallback 日誌，區分基礎設施錯誤與演算法錯誤
  • 論文主表格鎖定：補跑完成後生成最終 valid_record_matrix，凍結實驗參數，開始撰寫統計分析章節
  • 開源 evaluation harness：將 RAG_Ultimate 的多輪評測框架整理為可重用模組，支援其他研究者在相同穩健性標準下進行 RAG 比較研究

科學進步從來不只是「跑出更好的數字」，而是「以可信賴的方式，跑出可被他人複現的數字」。RAG_Ultimate 的 Tier-2 補跑，不只是為了讓 9/12 模式從「不可信」變「可信」——它更是在練習一種嚴謹的研究態度：對每一個異常都窮追到底，直到找到真正的根因。
