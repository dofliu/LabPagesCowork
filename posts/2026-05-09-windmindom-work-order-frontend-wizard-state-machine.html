一、為什麼工單系統是離岸風場 O&M SaaS 的核心？

離岸風場的運維管理，本質上是一個精密的「工單排程與追蹤」問題。當 SCADA 系統偵測到渦輪機異常、或定期維保計畫到期，都會觸發一張工單：確認問題類型、派遣技術人員、協調物料準備、安排離岸交通窗口（天氣、波高、CTV 排班），最終完成維修並記錄成本與工時。這個流程的複雜度遠超過一般辦公室的工作單。

傳統做法是靠 Excel 或通用工單系統（如 Salesforce Field Service、SAP PM）管理，但這些工具缺乏對離岸風能領域的深度語義理解：它們不知道什麼是「翻越波浪限制（Wave Height Limit）」、「鐵塔內部受限空間作業」、或「葉片吊掛 PTW（Permit to Work）」。windMindOM 的目標是打造一套真正懂離岸風場語境的運維工具——而工單模組是整個系統的運作中樞。

💡 離岸風場工單的特殊性

相較於陸上工業設備，離岸風機維修有幾個結構性複雜因素：首先是「天氣窗口限制」，CTV（Crew Transfer Vessel）作業的波高上限（通常 Hs ≤ 1.5m）讓工單必須等待合適的天氣窗口，這使得派工不能只考慮人力，還必須納入氣象預報。其次是「PTW（Permit to Work）安全作業許可」，高電壓/高空/受限空間作業都需要多階層授權。第三是「備料周期長」，部分關鍵備品的交期可達 3–6 個月，工單必須提前觸發採購流程。windMindOM 的工單狀態機設計，試圖將這些業務規則結構化為系統約束。

二、三步驟建立精靈（CreateWorkOrderWizard）

工單建立是使用頻率最高的操作之一，UI 設計的關鍵挑戰是：如何讓資訊填寫有秩序、不讓操作員面對一頁充滿欄位的表單？windMindOM 採用三步驟精靈（Wizard）模式，將工單建立流程拆解為邏輯上前後相依的三個階段：

Step 1 — 選擇風機

Farm 自動偵測 + 風機清單，點選目標風機及其目前狀態確認

Step 2 — 工單細節

工單類型（預防性 / 故障修復 / 安全巡檢）、優先級、描述、計畫日期

Step 3 — 派工與工時

指定主責技術員、預估工時（小時）、備注與特殊作業需求

這個設計有兩個工程上的考量。第一，步驟間的資料依賴是線性的：Step 2 的選項會根據 Step 1 選定的風機動態調整（例如已有開放工單的風機顯示警示）；Step 3 的派工清單則根據 Step 2 的工單類型過濾具備對應技能的技術員。第二，精靈模式允許使用者中途返回修改，且每步的資料持久在元件狀態中，不會因為切換步驟而遺失。

三、工單狀態機：7 個轉換表單的 DDD 設計

工單的核心複雜度在於其生命週期狀態機。每張工單從建立到結案，需要歷經多個狀態轉換，且每個轉換都需要不同的資訊輸入與授權驗證。windMindOM 遵循 Domain-Driven Design（DDD）原則，將工單狀態定義為領域物件的核心不變量，所有狀態轉換都只能通過明確定義的轉換函數進行。

OPEN
工單建立，待審核
ASSIGNED
已指派技術員
IN_PROGRESS
現場作業進行中
ON_HOLD
暫停（待備料/天氣）
PENDING_APPROVAL
等待主管審批
COMPLETED
作業完成，成本確認
CANCELLED
工單取消，附原因

前端實作了 7 個 inline 狀態轉換表單，全部嵌入 WorkOrderDetailModal，讓操作員在檢視工單詳情的同時就能執行狀態推進，不需要跳頁。每個轉換表單只顯示該轉換必要的輸入欄位——例如「開始作業（Start Work）」只需確認現場抵達時間；「暫停作業（Put on Hold）」需要填寫暫停原因和預計恢復時間；「完成（Complete）」需要填寫實際工時和維修摘要。

四、AbortController 與 Race Condition 防護

工業系統的前端有一個常被忽略但至關重要的問題：Race Condition。當使用者快速切換工單或重複點擊刷新，多個 API 請求會同時在途，後發的請求可能比先發的先回應，導致畫面顯示過期資料。

useWorkOrders.ts 在每次資料獲取請求中建立 AbortController，並在元件 unmount 或新請求發出時呼叫 abort()，確保只有最新的請求結果會更新狀態。這個看似小細節的防護，在生產環境中能避免操作員看到工單狀態「閃回」到舊狀態的困惑現象——在高壓運維情境下，這種視覺錯誤可能導致錯誤的作業決策。

💡 為什麼 DEV_ACTOR_ID 占位比 mock 更好？

目前工單的「執行者身份（actor_id）」使用固定的 DEV UUID 占位，等 M5 用戶身份認證模組完成後再替換。這個設計選擇反映一個工程哲學：使用真實的 API 流程走通整個工單生命週期，而不是用 mock 資料繞過它。DEV_ACTOR_ID 讓開發階段的測試能真實觸達後端資料庫，確認狀態轉換邏輯正確，之後的 auth 整合只需替換 actor_id 的來源，不需要重寫業務邏輯。相比之下，若使用 mock，auth 整合時往往會發現後端 API 設計與 mock 的假設存在落差，產生更大的重工量。

五、M3 里程碑進度與下一步

WMOM-20260504-19 完成後，M3 Milestone（工作流程 Part 1：工單 + 審批）的後端已 100% 完成（Issues -14 至 -18），前端工單部分（Issue -19）也已完成，整體 M3 進度達 90%。剩餘的 WMOM-20260504-20「審批前端（Approval Frontend）」——主管審批佇列的工作排程介面，是本里程碑的最後一塊拼圖。

從更大的產品路線圖來看，M3 完成後進入 M4（庫存管理 + 報表）。結合已完成的 M2 成本模組，windMindOM 的核心商業模組——工單派工、成本追蹤、審批授權——將形成可向第一個 PoC 客戶展示的完整閉環，是 2026-Q4 合約目標（NT$2–4M）的技術基礎。
