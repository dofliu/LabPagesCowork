研究程式碼的型別紀律：RAG Ultimate 如何用 6 輪 Routine 清除 TypeScript any 污染

2026-05-14 • 劉瑞弘教授團隊 • RAG Ultimate / TypeScript / Code Quality / Research Software Engineering

RAG Ultimate TypeScript Code Quality Research Software D3.js

2026 年 5 月 13 日，RAG Ultimate 的 Git 歷史新增了 6 個 routine PR——不是新功能，不是論文實驗，而是一場有計畫的型別清理行動。從早上第一輪消除後端公開介面的 ruff D 系列警告，到傍晚最後一輪為 D3 force-graph 的 callback 訂製區域型別，整整 6 輪下來，2,524 個測試全數通過，TypeScript 編譯器零錯誤。這不是運氣，而是一套刻意設計的研究程式碼品質工作流的體現。

學術研究的程式碼長期以來面臨一個兩難困境：快速驗證想法的壓力往往讓型別標註、文件字串、測試覆蓋這些「慢功夫」被一再推遲。結果是，三個月後自己都看不懂當初的邏輯，學生接手更是一頭霧水。RAG Ultimate 嘗試用「Routine 制度」打破這個循環——每個開發 session 分成功能衝刺與品質維護兩個模式，後者有專屬的 ROUTINE.md 清單，依優先順序逐項執行，每 5 個 PR 後強制做一次 CR/CI checkpoint。

六輪 Routine 做了什麼？

每一輪都有明確的任務邊界、受影響的檔案清單、以及下一輪的 next_hint。這種設計讓每個 PR 都獨立可 revert，也讓未來的協作者能從任一點切入而不失去上下文。

  • r1–r2 後端 9 個檔案 / 46 個 docstring 補全：針對 ruff D101/D103/D104 偵測出的公開介面缺漏，逐一補上 module-level、class-level、function-level 文件字串。禁區（services/）暫不動，保留供使用者授權後處理。
  • r3 後端 tests 目錄 / 29 個 docstring：測試類別與 fixture 的文件字串往往是最常被忽略的。本輪補完 test_chunking（13 個方法）與 test_vector_store（10 個方法），讓測試意圖在 ruff 掃描下完全透明。
  • r4 CR/CI Checkpoint（每 5 PR 強制執行）：全套跑 tsc / jest / ruff / pytest，回顧 PR #81–#84 的變更品質，確認無 regression，並將 2 個 pre-existing legacy 失敗標記為不相關。
  • r5 config 與 scripts 的殘留 docstring：ruff 重掃後發現禁區外還剩 4 處，包含 Settings.Config 巢狀類別（D106）與三個 main() 入口（D103）。一次清零。
  • r6 D3 force-graph 型別噪音清理：最技術性的一輪。GraphRagModal 的 D3 simulation callback 充斥著 (d: any) 與 event: any，用三個區域型別 SimNode / SimLink / DragEvent 全數替換。

D3 型別問題的深層邏輯

D3 的 force simulation 是 RAG Ultimate 的 GraphRAG 視覺化核心，但 @types/d3 尚未安裝的情況下，d3 的型別推回就是 any——這意味著每個 tick callback、drag handler 的參數都失去靜態檢查能力。最直覺的解法「全域 augment .d.ts」在這個情境下行不通，因為沒有可以 augment 的目標型別。

// 清理前：型別資訊消失，編譯器無法推斷 d 的形狀 forceLink(...).id((d: any) => d.id) node.attr("fill", (d: any) => colorMap[d.type]) // 清理後：區域型別明確標示 tick 後動態注入的屬性 type SimNode = GraphNode & { x?: number; y?: number; vx?: number; vy?: number; fx?: number | null; fy?: number | null } type SimLink = GraphLink & { source: string | SimNode; target: string | SimNode } forceLink(...).id((d: SimNode) => d.id) node.attr("fill", (d: SimNode) => colorMap[d.type])

這個設計有一個特別值得記錄的工程決策：選擇區域型別而非強行安裝 @types/d3。原因是引入型別套件屬於 devDependency 變更，需要使用者確認；而區域型別在不改動任何 runtime 行為的前提下，已能完整標注 callback 的意圖，且日後若決定引入 @types/d3，直接替換為官方的 SimulationNodeDatum / SimulationLinkDatum / D3DragEvent 即可，migration cost 為零。

🔍 什麼是「禁區」概念？

RAG Ultimate 在 Routine 流程中明確定義了「禁區」（off-limits zones）——通常是核心服務層（services/embeddingService.ts、services/geminiService.ts等），這些檔案的變更風險高、影響範圍廣，每次修改前都需要使用者明確授權。這個機制讓 AI 協作開發在「積極清理」與「謹慎保守」之間取得平衡——能自動處理的就自動處理，需要人類判斷的就停下來等待確認，而不是偷偷修改核心邏輯。

Routine 制度：研究室程式碼的「定期健檢」

傳統學術軟體開發的典型死亡螺旋是：功能越堆越多 → 技術債越積越深 → 新成員上手成本越來越高 → 更沒有時間清債。RAG Ultimate 的 Routine 制度試圖在源頭打斷這個循環，核心設計原則有三點：

  • 每輪有明確的任務邊界：不做「順手修一下」的模糊改動，每個 PR 的 diff 都應該能在三行以內解釋清楚目的。
  • next_hint 是跨 session 的記憶體：每輪結束時，在 routine_log.yaml 記錄下一步的候選項目，讓下個 session（或隔天、隔週）的開發者能無縫接手，不需要重新分析程式碼狀態。
  • 每 5 個 PR 強制 checkpoint：避免多個「安全」的小改動在累積後出現意外的交互作用。checkpoint 不只跑 CI，還做人工的 diff review，確保方向正確。

💡 研究程式碼品質的本質矛盾

學術研究需要快速探索、頻繁重構、有時甚至需要「先讓它跑起來」的原型思維；但同時，研究程式碼又是論文可重現性的基礎，需要比商業軟體更嚴格的文件與測試覆蓋。RAG Ultimate 的做法是把這兩種模式明確分開：功能衝刺時允許暫時的型別妥協，但 Routine session 是有計畫地清理這些技術債，而不是讓它們永遠留在 codebase 裡。關鍵在於：清理的節奏要快過技術債積累的速度。

對研究室同學的啟示

不管是做 RAG 系統、工業 AI 或風能故障診斷，當你的研究程式碼規模超過幾千行之後，型別標注與文件字串不再只是「好習慣」，而是認知負荷管理的基礎設施。當你三個月後回來改一個 bug，或者學弟妹要接手你的程式碼，清楚的型別定義和 docstring 能省下大量的逆向工程時間。

幾個可以立刻採用的具體做法：

  • 在 Python 專案中加入 ruff --select D101,D103,D104 到 CI pipeline，讓公開介面的文件字串缺漏在 PR 階段就被擋下。
  • TypeScript 專案開啟 "noImplicitAny": true，把 any 的使用限制在有意識的決策，而不是偷懶的逃脫出口。
  • 為每個外部函式庫缺少型別定義的情況，建立「最小區域型別」而不是直接用 any——就算只是 type D3Node = { id: string; x?: number; y?: number } 也比什麼都沒有好。
  • 維護一份 routine_log.yaml，每次做完維護工作後記錄 next_hint，讓自己和協作者能快速恢復上下文。

🚀 RAG Ultimate 下一步

在型別清理完成後，下一個重大里程碑是 Stage 2B ResearchLab 重組——把實驗計畫（Experiment Plan）與即時遙測（Live Telemetry）重組為雙欄佈局，並補跑 r06/r07 實驗輪次，積累足夠的完整 run 數量以支撐論文主表的統計分析。目前整個平台已通過 829 個測試（41 個測試套件），17 種 RAG 模式穩定運行，Qdrant 向量資料庫持久化正常，這些基礎條件讓研究團隊能夠更有信心地設計大規模評測實驗。

一個研究平台的成熟度，不只體現在它能做多酷的功能，也體現在它能以多穩定的節奏自我更新。RAG Ultimate 的 6 輪 routine 是一個縮影：好的研究程式碼需要持續的打磨，而打磨需要制度而不只是善意。
