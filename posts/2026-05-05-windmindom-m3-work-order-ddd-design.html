windMindOM M3 設計週：從現場系統逆向工程到領域驅動設計

2026-05-05 • 劉瑞弘教授團隊 • windMindOM / 離岸風場運維 / DDD / Work Order

windMindOM Domain-Driven Design Work Order 離岸風場 O&M

開發一套全新的離岸風場運維（O&M）管理平台，最難的地方往往不是技術選型，而是弄清楚業務流程本身究竟是什麼。台灣目前的陸上風場維護，許多廠商已有多年運作的現場管理系統——邏輯複雜、歷史包袱深，但同時也蘊含著真實業務場景的寶貴知識。windMindOM 的 M3 里程碑，花了一整個設計週做一件事：把既有的現場系統（陸上版）徹底逆向工程，提取其中的業務語意，再用現代的領域驅動設計（Domain-Driven Design, DDD）原則重新打造。

這不只是一次技術重構，更是一次「從看懂別人怎麼做，到用更好的方式重做」的過程。本文記錄 DN-01（工單生命週期）、DN-02（多層簽核）、DN-03（庫存領料）三份設計筆記背後的核心洞察，以及 windMindOM 如何在離岸情境下超越陸上原型。

一、逆向工程的挑戰：讀懂 1,600 行 if 判斷

windMindOM 的設計起點是一套真實使用中的陸上風場 CMS，基於 Node.js + MongoDB 架構，由現場工程師多年累積而成。它能運作，但代碼結構反映了「邊做邊加功能」的典型演進路徑：一張工單的生命週期散落在 repairTemp、repair、trackFrom、removeFrom 四個 MongoDB collection 之間；多層簽核邏輯被切成 employeesign、leadersign、supervisorsign、affairsign 四個獨立 collection；前端需要 1,600 行以上的 if 判斷才能正確解讀工單當前狀態。

逆向工程的核心任務是：在這些「可運作但難維護」的代碼背後，找到真正的業務語意。以簽核流程為例，最令人驚訝的發現之一是：etech 系統中「工單簽核完成」的表達方式，是把簽核紀錄文件（document）直接刪除（DELETE）——當某個 collection 內的 doc 消失，代表「已簽完」。這在短期內可以運作，但無法產生任何 audit trail，也無法回答「誰在幾點簽核了什麼」這個問責問題。

🔍 識別出的系統性反模式（Anti-patterns）

反模式 影響 windMindOM 對策
4 個 collection 表達 1 張工單 前端 1,600 行 if 判斷，狀態不可見 單一 work_order 表 + 7 態狀態機
「簽完」= DELETE doc 零 audit trail，法規遵循困難 status 欄位永不刪，加 event log
4 個並行 sign collection 改業務邏輯要動 4 處，無法串聯 SignoffChain + SignoffStep 串鏈模型
String 欄位連結人員 改名 = 全 collection updateMany，無交易 surrogate UUID FK，不依字串比對
庫存 PUT 整份 doc 改數量 並發競爭，無入庫出庫明細帳 event-sourcing 出入庫流水帳 + 4 quantity 欄位

二、DN-01 核心設計：7 態狀態機取代 4 個 Collection

工單（Work Order）是整個 O&M 系統的核心業務物件。DN-01 的設計決策，是以單一資料表 + 明確狀態機取代原本散落四處的 collection 架構。windMindOM 的 WorkOrderStatus 定義了 7 個狀態節點：DRAFT（草稿）→ DISPATCHED（已派工）→ IN_PROGRESS（進行中）→ AWAITING_SIGNOFF（等待簽核）→ CLOSED（關閉），以及側邊的 CANCELLED（取消）與 REOPENED（重開，對應「需改善」後續追蹤場景）。

每一個狀態 transition 都有明確的 Guard 條件與觸發 actor：例如 start_work 從 DISPATCHED 到 IN_PROGRESS，在離岸場景下必須額外確認 vessel_id（維修船 ID）與 weather_window_id（天氣窗口 ID）已綁定——這是陸上系統完全不需要的約束，但在離岸風場卻是攸關工安與成本的關鍵管控點。

🏗️ windMindOM WorkOrder 四大設計原則

  • Surrogate UUID 為主鍵：所有 FK 與 API 用 UUID，WO-Z72TC-202607-01 格式的 business key 只給人類（UI / 報表 / 列印）閱讀，底層不依賴。
  • Progress notes append-only：維修人每次更新狀況用 append（追加紀錄）而非 PUT 就地覆寫，完整保留維修過程的時序紀錄。
  • Event log 取代 allForm audit copy：etech 在完工時把整份 doc 複製進 allForm（無時間排序保證），windMindOM 改為輕量 event-sourcing——每次狀態 transition 寫一筆 WorkOrderEventLog，記錄 from/to status、actor、timestamp 與 payload。
  • 離岸延伸欄位預留但可空：vessel_id、weather_window_id、logistic_hours 在陸上情境下皆為 null，不影響陸上系統使用，但離岸部署時 start_work Guard 會強制要求填入。

三、DN-02 簽核鏈：從 Fan-out 刪除到 Chain Step 模型

多層簽核是工業 O&M 系統中不可或缺的合規機制，也是 etech 系統設計最混亂的部分之一。etech 的做法是「fan-out 並行 sign」：當一張工單完工，前端同時往 4 個不同 collection 各塞一份 pending sign 文件；「簽完」等於把 doc DELETE——這意味著沒有任何機器可讀的「核准紀錄」。

更關鍵的問題是：這套設計無法支援順序化簽核需求。在離岸風場的實際運維中，主管批准工單之後，倉管人員才能撥料——etech 的並行 fan-out 架構根本無法表達這種先後順序。windMindOM 的 DN-02 設計以 SignoffChain（簽核流程定義）+ SignoffStep（每一步的簽核狀態）取代四個並行 collection，每個 step 的狀態為 pending / approved / rejected / skipped，doc 永不刪除，全留 audit trail；支援串聯（sequential）與並聯（parallel）混合的簽核流程；reject 動作可帶 comment 欄位並自動把工單退回 IN_PROGRESS 狀態。

💡 為何 Audit Trail 在離岸風場特別重要

離岸風場涉及海事作業安全規範與保險理賠，任何一張工單的「誰批了什麼、何時批的」都可能成為事故調查的關鍵證據。台灣的離岸風場開發案通常由國際開發商主導，歐洲 DNV 或 Bureau Veritas 等驗證機構對 O&M 系統的 traceability 有明確要求。windMindOM 的 event log + signoff audit 設計，不只是工程上的「好習慣」，更是未來進入國際市場的合規基礎。

四、離岸延伸：天氣窗口與成本核算的前置設計

windMindOM M3 設計最與眾不同之處，是在每個設計決策中都刻意預留了「離岸延伸軸線」。陸上風場的工單可以隨時派工；離岸風場則受到海況、天氣窗口（Weather Window）的嚴格限制——當波高超過 1.5 米或風速超過閾值，CTV（crew transfer vessel）無法安全靠岸，工單派工必須等待窗口。這個約束，直接影響了 O&M 成本。

windMindOM M2 完成的 ECN cost engine 已能計算「因等待天氣窗口產生的停機損失」；M3 設計的工單 schema 在 closed 事件時自動 emit 4 種成本帳目：人工成本、料件成本（來自 DN-03 的領料單）、船隻費用（logistic_hours × vessel.day_rate），以及發電損失（從警報發生到工單關閉的停機時間 × 風機預期發電量 × 電價）。這 4 個數字會自動寫進 M4 的 cost ledger，提供「預測 vs 實際」的對比分析——正是 ECN K13 成本引擎設計時預留的「actual」欄位。

windMindOM M3 的設計週展示了一種有效的開發策略：先逆向工程理解現有業務邏輯，再用現代設計模式重新組織。這不是「重寫主義」（rewrite from scratch 往往失去業務知識），而是「業務語意提取 + 技術實現替換」。對陸上風場的運維邏輯（誰開單、誰審批、領什麼料）保持高度尊重，同時系統性地清除技術反模式，並預留離岸情境所需的額外維度（天氣窗口、船隻調度、跨國合規）。這個方法，也許正是 DOF Lab 在台灣離岸風電 O&M 數位化這條路上，能夠與陸上系統既有廠商差異化競爭的核心能力之一。
