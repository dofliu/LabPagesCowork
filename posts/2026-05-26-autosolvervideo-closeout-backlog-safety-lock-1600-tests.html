Safety Lock 工程方法論：autoSolverVideo Closeout Backlog 衝刺，iter 119–134，1600 條測試全綠

2026-05-26 • 劉瑞弘教授團隊 • EdTech / Software Quality / Test Engineering / Production Hardening

autoSolverVideo 1600 Tests Safety Lock Closeout Backlog FastAPI / pytest / CI

生產級系統有一種特殊的技術債，比功能缺失更隱蔽、更危險： 功能已經跑了幾個月、一直沒問題，但從來沒有直接的測試保護它。 任何一次重構、一個不起眼的變數名稱調整，都可能無聲無息地改變行為， 直到用戶回報奇怪的輸出你才意識到問題已經上線了。

DOF Lab 的教學影片自動生成平台 autoSolverVideo 在過去兩天（2026-05-24 至 05-25） 展開了一場針對這類技術債的系統性清除行動——「Closeout Backlog」。 這不是修 bug，也不是加新功能：它是一套稱為 Safety Lock 的工程方法論， 目標是為所有「裸奔中的 production 函式」補上行為鎖定測試，確保任何未來的重構 都有護欄可以依靠。從 iter 119 的 1,397 條測試出發，到 iter 134 的 1,600 條， 兩天之內新增 203 條，CI 四組矩陣（Ubuntu/Windows × Python 3.10/3.12 + frontend typecheck）全綠。

一、什麼是 Safety Lock？為什麼不叫 TDD？

傳統的測試驅動開發（TDD）是「先寫測試，再寫功能」。但對一個已上線、已被用戶信賴的 production 系統而言，這個順序是反過來的。autoSolverVideo 的許多核心函式—— _run_ingest_repo、 _run_render_phase、 _merge_sections_to_final—— 從最初的 PR 就在跑，一路迭代新增功能，從未有對應的直接單元測試。

Safety Lock 的核心哲學是：「不改 production code，只為它的行為立下契約」。 每一條測試都是一份合約：這個函式收到什麼輸入，應該產出什麼輸出，遇到例外時應該 如何降級（graceful degrade）而非炸掉整個任務。一旦契約存在，未來的重構就有了 可以安心依靠的護欄——任何行為漂移都會立刻被 CI 抓到，而不是在幾週後的用戶回報中才浮現。

🔒 Safety Lock 三原則

  • 0 production code 改動：每個 iter 只加測試，不動功能程式碼。這避免了「修測試順手改了邏輯」的風險。
  • 行為意圖文件化：測試本身就是設計文件。例如「mermaid 抽取失敗應吞例外不擋 ingest 主流程」這樣的 design intent，現在被測試永久保存。
  • 對比鎖（Symmetric Locking）：對有對稱邏輯的函式（如 intro/outro、image/mermaid 兩條生圖路徑），明確鎖住「這兩條路徑日誌訊息不同、行為對稱但不相同」，防止重構時被合併成同一條。

💡 一個意外的收穫：揪出 non-str title bug

Safety Lock 最有趣的副產品，往往不是測試本身，而是「寫測試時發現的問題」。 iter 121 在補 _read_deck_title 的邊角測試時發現： 若 deck 頂層回傳 list 或 title 欄位是 int， 函式會在 .strip() 呼叫時炸出 AttributeError， 影響整個 Library 頁面的渲染。由於 Gemini 在特定情境下確實可能產生結構偏差， 這是一個真實的生產風險。依據 Safety Lock 硬規則，此 bug 已記錄進 docs/ROUTINE_FINDINGS.md，等待下一個 session 正式修復。

三、為什麼這對研究型工程團隊特別重要？

學術研究室的軟體開發有一個典型困境：快速迭代驗證想法和 穩定可靠的系統品質之間的張力。autoSolverVideo 從一開始就是 一個「快速驗證教學影片自動化」的研究工具，功能一個接著一個加進去， 每一個 iter 都要交付可運行的成果，很難每次都先寫測試。

Closeout Backlog 提供了一個「補票」的系統性路徑：功能開發衝刺與測試補全衝刺 在時間上分離，各自有明確的節奏。功能衝刺讓研究快速推進；測試補全衝刺 讓系統在累積夠多功能之後，能夠安全地進行更大規模的重構，而不必擔心悄悄引入回歸。 對於任何走向 production 的研究系統，這套「先做後鎖」的節奏，或許比教科書式的 TDD 更符合研究室的實際工作流。

⚠️ 「裸奔的 production 函式」的真實代價

iter 123 在補 themes.py 的主題順序契約時，揭示了一個無聲 bug 的存在路徑： frontend 的 THEME_OPTIONS 和 backend 的 THEME_LIST 分別寫死了 15 個主題的順序， 兩者應該完全一致，但從沒有自動化驗證。若有人改了 backend 順序忘了 更新 frontend，用戶選的是 forest 風格但渲染出來的卻是 navy——這種靜默錯誤 不會炸程式，只會讓用戶困惑，而且很難 debug。Safety Lock 把這種「隱式契約」 變成了「顯式測試」。

四、下一步：Closeout Backlog 2/4 到 4/4

iter 134 標誌著 Closeout Backlog 第一階段（1/4）的完成。 接下來的三個階段已經規劃好：

  • iter 135（Closeout 2/4）：_run_ingest_long_form 測試覆蓋——處理 document 和 URL 來源、PDF figures 抽取、outline_long_form / script_long_form 透傳

  • Closeout 4/4：E2E 整合測試，從 API call 到 artifacts 輸出的完整 smoke test，為日後的重構和新功能開發提供最後一道安全網

1,600 條測試是一個數字，但它背後代表的是 1,600 個明確的行為契約—— 每一個都是未來重構的護欄，每一個都是這個系統「我知道自己在做什麼」的聲明。 對一個以研究為驅動、以工具為產出的實驗室而言，這種品質基礎設施的建立， 和論文產出、功能開發一樣，是核心競爭力的一部分。
