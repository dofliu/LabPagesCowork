RAG API 的第一道防線：Schema 型別邊界、Metadata Filter 與 Score Threshold 設計

2026-05-18 • 劉瑞弘教授團隊 • RAG / API Design / Schema Validation / Research Engineering

RAG_Ultimate API Design Schema Validation Pydantic Research Engineering

在設計 RAG（Retrieval-Augmented Generation）研究平台時，多數研究者的注意力自然集中在「檢索效果」本身：哪種分塊策略最好？向量模型怎麼選？Hybrid Search 的權重如何調？但有一個問題常被低估——壞資料在進入檢索管道前，應該在哪裡攔截？

RAG_Ultimate 今日完成了兩個相互呼應的 API Schema 強化：PR #105（IngestRequest）與 PR #106（RetrieveRequest）。這兩個 PR 都只觸碰 schemas.py 與對應測試，完全不動核心向量存取與檢索邏輯——但它們的意義遠超過「加幾個欄位驗證」：它們代表的是以 Schema 作為研究平台第一道防線的設計哲學。

問題的起點：無邊界的自由字串

RAG API 的輸入通常看起來很簡單：給我一個 collection_name，一個 query，還有 top_k，我幫你找最相關的文件。問題在於，如果這些欄位沒有任何邊界約束，就可能收到這樣的請求：

# 問題場景一：空白 collection_name 流到 Qdrant 才報錯 { "collection_name": "", "query": "風力發電故障診斷", "top_k": 10 } # 問題場景二：超長 query 把 embedding API 餵成 batch { "collection_name": "wind_papers", "query": "（2000 字的完整論文段落...）", "top_k": 5 } # 問題場景三：collection_name 含斜線，Qdrant 路由混亂 { "collection_name": "wind/papers/v2", "query": "gearbox vibration", "top_k": 3 }

在沒有 schema 邊界的情況下，這些問題通常在服務的最深層——向量資料庫或 embedding 模型——才會以奇怪的錯誤訊息浮現，讓 debug 成本大幅上升。

PR #105 + #106：兩層 Schema 的互補強化

本次強化以 Pydantic v2 的宣告式驗證為基礎，分別針對資料注入（Ingest）與資料檢索（Retrieve）兩個方向補上邊界。

📌 Schema 強化對照表

欄位 強化前 強化後 防範問題
collection_name（兩者） 任意字串 min_length=1, max_length=128, pattern 限制 空白名稱 / 含 / 等特殊字元流到 Qdrant 層才報錯
query（Retrieve） 任意字串 min_length=1, max_length=2000 超長文本把 embedding API 誤當成批次送入
query_filter 新增 不存在 Optional[Dict[str, Any]] 為 Qdrant payload filter pass-through 預留型別位置
score_threshold 新增 不存在 Optional[float]（ge=0.0, le=1.0） cosine similarity 超出 [0, 1] 範圍被攔在入口

值得注意的是，query_filter 與 score_threshold 這兩個新欄位在這個 PR 中尚未串接至向量存取層——它們只是在 Schema 上預留了型別位置。這個設計決策並非偷懶，而是刻意的：

🔒 禁區制度（Safe Zone）
RAG_Ultimate 的例行維護有嚴格的「禁區」規範：核心 vector_store 與 embedding_service 邏輯不得在未計畫的情況下觸碰。Schema 強化只動 schemas.py，確保每個 PR 的影響範圍清晰可控。
📐 型別先行策略
先在 Schema 建立正確的型別宣告，後續的 wire-up PR 才有清楚的 interface 可以對接。這種「contract first」的做法讓後端邏輯和 API 邊界解耦，降低重構成本。

Metadata Filter 在 RAG 研究中的意義

query_filter 的加入，背後有更深的研究動機。RAG 系統最常見的問題之一是語意相關但情境不符：使用者查詢「齒輪箱振動異常」，向量搜尋確實找到了語意接近的文件，但那些文件來自 2018 年、針對不同型號的離岸風機，對當前的診斷任務毫無幫助。

Qdrant 的 payload filter 機制允許在語意搜尋之外，額外限定文件的 metadata 條件——例如指定文件年份、來源資料集、設備型號或風場地點。在 RAG_Ultimate 的研究框架中，這個能力對我們正在進行的風場文件 RAG Benchmark 論文特別重要：不同品質、不同來源的文件混入同一個 collection 時，metadata filter 就是確保評測可控的關鍵工具。

💡 研究啟示：Schema 是 RAG 系統的「型別契約」

Pydantic Schema 在 RAG 平台中不只是「資料驗證工具」，它更像一份公開的型別契約：告訴呼叫端這個端點能接受什麼、不能接受什麼、未來會提供什麼能力（query_filter、score_threshold）。這份契約愈早建立、愈完整，後續添加功能時需要的「溝通成本」就愈低——無論呼叫端是人類工程師、前端元件，還是 AI 代理。

對於 AI 輔助研究平台來說，這一點尤其重要：當自動化例行程序每天數十次觸發 API 呼叫時，清晰的 Schema 邊界是防止靜默失敗（silent failure）累積成資料品質危機的最後一道閘門。

21 個新測試：驗證邊界的邊界

PR #106 新增了 5 個測試 class、21 個測試案例，涵蓋所有新舊欄位的邊界行為。測試設計遵循「正確路徑 + 邊界合法值 + 邊界非法值」三層結構：

# 示例：score_threshold 的邊界測試（6 個案例） class TestRetrieveRequestScoreThreshold: def test_valid_none(self): ... # None → 向後相容 def test_valid_zero(self): ... # 0.0 → 最寬鬆閾值 def test_valid_one(self): ... # 1.0 → 僅完全相符 def test_valid_mid(self): ... # 0.75 → 典型使用值 def test_invalid_negative(self): ... # -0.1 → 攔截 def test_invalid_above_one(self): ... # 1.1 → 攔截 # 合計測試通過：86 passed（65 既有 + 21 新增） # ruff check：All checks passed # npx tsc --noEmit：0 errors

完整的測試覆蓋也是這個「禁區外」策略得以成立的前提：當我們保證 schema 層的行為完全可預測，才能安心地在後續 PR 中進行 wire-up，而不用擔心引入隱性的輸入假設。

連續等待：PR #106 的 5 連 Skip 現象

有一個值得記錄的工程現象：PR #106 自昨晚建立後，例行品質電池今日已連續觸發 5 輪排程，全部因「等候 PR merge」而跳過執行。這與歷史最高紀錄（PR #99 時的 5 連 skip）並列。

系統的行為完全如預期：每輪 log 忠實記錄目前狀態、提示建議的人工介入指令，但絕不自行 merge。這種「能見度優先、不擅越界」的設計正是 AI 輔助研究工作流的核心原則——自動化應放大研究者的生產力，而不是繞過決策責任。PR #106 CI 全綠、與既有程式碼向後相容、無任何待解 review，等候的唯一原因是人類尚未批准 merge。
