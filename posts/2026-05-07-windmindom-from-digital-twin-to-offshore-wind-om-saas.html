從數位分身到離岸風場運維 SaaS：windMindOM 的商業化路徑解析

2026-05-07 • 劉瑞弘教授團隊 • windMindOM / 商業化策略 / 離岸風場 O&M / SaaS

windMindOM 離岸風場 O&M 商業化 SaaS 策略 PoC

一套能通過 121 項物理驗證測試的數位分身，距離「讓運維廠商願意付錢購買」，還差幾步？這是 windMindOM 專案在完成 WMOM-23（7 層 Physics 自我驗證框架）後，正面對的核心商業問題。本文從產品與策略視角，解析 windMindOM 目前的進展狀態、六個月路線圖，以及達成第一筆合約目標（NT$2–4M）的關鍵路徑。

一、windMindOM 是什麼？從學術原型到商業產品的演化

windMindOM 的起點是 DOF Lab 的另一個研究系統 digiWindTurbine——一套以物理模擬為核心的離岸風機數位分身平台，包含 104 個 SCADA 標籤、11 種故障情境模擬、Bastankhah–Porté-Agel 尾流模型，以及大氣穩定度五重耦合計算。這套系統在學術上已有相當完整的技術基礎，但在商業化面向的缺口是顯而易見的：運維廠商需要的不只是模擬器，而是一套整合監控、派工、成本計算、報表、知識查詢的日常作業工具。

windMindOM v0.8.1 便是這個商業化升級的產物。它繼承了 digiWindTurbine 的全部物理模擬底層，在上方疊加五個運維業務模組：即時監控（Monitoring）、工單派工（Workflow / Work Order）、成本計算（Cost）、報表（Reporting）、知識查詢（Knowledge / RAG）。技術棧採用 FastAPI + React + SQLite + Modbus TCP + OPC UA（Bachmann Z72），並以 Docker Compose 封裝，目標是可以直接在客戶現場或雲端部署。

💡 為什麼選擇 SaaS 而非單次授權？

windMindOM 的商業模式設計為 SaaS（Software as a Service）訂閱制，而非傳統的軟體一次性授權。核心原因有三：首先，離岸風場的運維資料具有高度時序性，需要持續更新的模型和知識庫，單次授權無法支持這種動態維護需求。其次，SaaS 模式能讓開發團隊持續收到真實使用回饋，從而快速迭代改善產品。第三，從財務角度看，訂閱制帶來可預期的月經常性收入（MRR），比一次性合約更適合支撐後續研發投入。

二、六個月路線圖：技術優先順序的戰略設計

windMindOM 的六個月路線圖（M1–M6）在設計上有一個關鍵邏輯：先讓產品可信賴，再讓產品易使用，最後讓產品能賣出。這個順序不是偶然的。

里程碑 時程 內容 狀態
M1 2026-05 基礎建設 + 客戶接觸 infrastructure 進行中（85%）
M2 2026-06 成本計算模組（ECN 移植，4 個引擎） ✅ 完成（100%）
M3 2026-07 工作流程 Part 1（工單 + 簽核審批） 進行中（80%，後端完成，前端待接）
M4 2026-08 工作流程 Part 2（庫存）+ 報表模組 計畫中
M5 2026-09 Knowledge / RAG + 現場行動 UI 計畫中
M6 2026-10 第一個運維廠商 PoC + 第一筆合約 目標

M2 成本模組的完成是一個值得關注的里程碑：整套 ECN 成本計算框架（等待時間引擎 waiting_time、成本計算 cost_cal、蒙地卡羅 LCOE 模擬 monte_carlo、維修方案波動分析 var_fluct）在一天之內完成移植，並通過 bit-perfect 驗證（K13 基準：LCOE = 72.94 EUR/MWh）。這不只是技術成就，也是對未來客戶的一個重要訊號：windMindOM 內建的成本計算能力來自 ECN（荷蘭能源研究中心）的方法論，有學術背書。

三、「可信賴優先」工程哲學的商業意義

在 M3 後端完成後，原本合理的下一步是推進前端工單介面（WMOM-19）。但劉瑞弘教授做出了一個關鍵決定：暫緩 UI 開發，先完成物理模型的 7 層自我驗證框架（WMOM-23）。這個決定在產品開發的常規邏輯中看似「繞遠路」，但從商業化角度卻非常有遠見。

離岸風場運維是一個高風險、高合規要求的產業。一套決策輔助系統若無法向客戶提供「你的預測結果是基於符合物理定律的模型」的保證，就很難進入採購評估流程。WMOM-23 完成後，windMindOM 可以明確聲明：

  • 風機功率係數（Cp）被自動驗證不違反 Betz 極限（Cp ≤ 0.593）
  • 全系統能量守恆（氣動 × 傳動 × 轉換 ≈ 電力輸出，容差 ±5%）
  • 紊流譜符合 IEC 61400-1 Kaimal 模型
  • 11 種故障注入情境的 SCADA 物理 signature 均通過自動驗證
  • 每次測試紀錄永久保存，支援 baseline diff 與趨勢追蹤

在任何企業 IT 評估中，這些陳述都是重要的「可信賴性佐證」，可以直接進入 RFP（Request for Proposal）或 PoC 評估報告。先把「為什麼你應該信任我們的數字」說清楚，再討論「我們的介面長什麼樣」，這個順序對 B2B 銷售尤其關鍵。

🔍 目標客戶輪廓：台灣離岸風場運維廠商

windMindOM 的第一波目標客戶是台灣本地的離岸風場運維服務提供商（O&M Service Provider）。台灣目前已裝置的離岸風機超過 300 支（大彰化、海能、風睿等），且根據 2035 能源政策，還有大量裝置計畫在推進。這些風機的運維管理，目前多依賴 Excel、email 與人工協調，系統化工具的需求空缺相當明顯。windMindOM 的第一個 PoC 合約目標（NT$2–4M）瞄準的正是這個市場缺口：一套結合物理模擬底層的運維 SaaS，比純前端工具更難被競爭對手複製，且能直接對接現場 OPC UA / Modbus TCP 資料。

四、下一步：前端工單介面與 Z72 現場整合

在 WMOM-23 完成、物理基礎夯實之後，windMindOM 的當前開發焦點回到 M3 的最後一哩路：前端工單介面（WMOM-19）與工單審批前端（WMOM-20）。後端 REST API 已完整實作（Work Order CRUD、Approval signoff、狀態機），剩下的是將這些功能以直觀的 React UI 呈現給現場運維人員使用。

同步推進的還有 M1 的客戶接觸 infrastructure。這一部分在技術層面相對簡單（主要是 demo 環境設定、文件準備），但在商業化節奏上至關重要：要在 M6（2026-10）達成第一個 PoC，客戶識別和關係建立必須在 M3–M4 階段同步展開，而不是等產品完成後才開始。

🎯 研究室當前行動項目

  • WMOM-19：前端工單介面 React 元件開發（M3 最後一哩）
  • WMOM-24：Data quality 修正（Layer 4 spread / CV 異常自動驗證已可 catch 問題所在）
  • M1 客戶接觸：準備 PoC 提案文件 + demo 環境，識別第一個潛在客戶
  • Z72 整合文件（DN-02 / DN-03）：Bachmann Z72 HMI 與 windMindOM backend 的設計藍圖已完成，實作待啟動

windMindOM 正站在一個關鍵的轉折點：技術原型已達到可以「說服工程師」的成熟度，下一個挑戰是達到可以「說服採購決策者」的商業化完整度。這需要的不只是更好的程式碼，還需要一套清晰的產品故事、可示範的 demo 路徑，以及足夠的業務觸點讓潛在客戶知道這套系統的存在。接下來六個月，技術與商業兩條線同步推進，將是 windMindOM 最重要的衝刺。
