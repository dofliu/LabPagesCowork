物理模型不能只是「採用」：windMindOM 的 7 層 Physics 自我驗證框架

2026-05-06 • 劉瑞弘教授團隊 • windMindOM / 物理模型驗證 / Digital Twin / 離岸風場

windMindOM Physics Validation Digital Twin 離岸風場 O&M WMOM-23

「物理模型不能只是說有採用，要知道結果是否準確。」

這句話來自劉瑞弘教授在 2026-05-05 的 windMindOM 開發 review，是一個讓整個 M3 衝刺方向出現重要轉折的決定。原定下一步是推進 M3 的前端工單介面（WMOM-19），但在 code review 之後，教授做出了一個反直覺的選擇：暫緩 UI 開發，先回頭強化物理模型的驗證機制。這個決策背後，蘊含著一套關於「數位分身可信度」的深刻工程哲學。

一、數位分身的核心矛盾：你怎麼知道它是對的？

windMindOM 的物理模擬核心繼承自 digiWindTurbine 計畫，涵蓋範圍相當完整：104 個 SCADA 標籤、Bastankhah–Porté-Agel Gaussian 尾流模型、大氣穩定度五重耦合（shear / TI / wake / veer / L_u）、疲勞損傷 Miner 法則、剩餘壽命（RUL）預測，以及 11 個故障情境（包含葉片結冰、主軸承磨損、齒輪箱過熱等）的物理耦合模擬路徑。

問題在於：這些模型「有在跑」，但從未被系統性地驗證過「跑出來的結果是否物理上合理」。目前唯一的自動檢查機制是一個半自動的 21 項 data quality check 腳本（examples/data_quality_analysis.py），且其中 3 項已知 fail（個體差異 spread 異常、Region 3 功率 CV 偏高）。

⚠️ 既有物理模型的四大驗證空白

  • 未驗證物理定律與文獻 benchmark：Betz 極限（Cp ≤ 0.593）、IEC 61400-1 Kaimal 紊流譜、ISO 10816-3 振動標準是否被滿足，沒有自動化測試確認。
  • 未驗證故障注入 sanity：「注入軸承磨損故障後，高頻振動應該上升」這個基本物理預期，沒有任何自動驗證會 catch 到異常。
  • 未驗證跨模組一致性：P_aero × η_drivetrain × η_converter ≈ P_elec 這條能量守恆約束，沒有任何測試會在鏈路斷裂時發出警告。
  • 「憑感覺」的修改流程：修改物理常數後，只能靠肉眼觀察儀表板來判斷結果，邊角 case 完全靠手動發現。

這不只是測試覆蓋率的問題，更是數位分身可信度（trustworthiness）的根本問題。若一套風機 digital twin 無法證明自己遵守 Betz 定律，它就無法取得研究界或業界的信任。這是 WMOM-23 被列為 P0 critical 的原因。

二、7 層 Physics Validator：從守恆律到測試紀錄

WMOM-23「Physics 自我驗證框架」設計了一套 7 層遞進式驗證體系，每一層解決不同維度的物理可信度問題，且每層獨立 commit、可逐步交付。

層 名稱 核心驗證目標
Layer 1 守恆律 & 物理上限 Betz 限（Cp ≤ 0.593）、能量守恆、動量平衡、熱平衡、質量守恆
Layer 2 文獻 Benchmark IEC 61400-1 Kaimal 紊流譜、Bastankhah wake 公式、Glauert NTF、BPFO/BPFI 軸承頻率、ISO 10816-3 振動標準
Layer 3 操作邊界 切入/切出風速、緊急停機邏輯、Pitch/Yaw 速率限制、超速保護觸發條件
Layer 4 跨模組一致性 spread / CV / cubic R² / 溫度 lag / 大氣穩定度「五重耦合」驗證
Layer 5 故障注入 Sanity 11 個故障情境各自的 SCADA signature 自動驗證（如注入齒輪箱過熱後，oil_temp 必須上升）
Layer 6 一鍵體檢 CLI tools/physics_health_check.py：彙整前 5 層結果 + 自動產出 matplotlib 圖表
Layer 7 ⭐ 測試紀錄保存 tests/physics/reports/ 結構 + YAML metadata + baseline diff + retention policy（教授強調「每次測試留紀錄」）

其中 Layer 7 被標記為特別重要（⭐），因為它體現了教授的核心要求：不只是「跑完不報錯」，而是「每一次物理驗證的結果都要被記錄下來，日後可以追蹤趨勢、比對 baseline 差異」。這個思維與軟體工程中的 regression test 類似，但針對的是物理正確性而非程式行為——這是一個在 digital twin 領域相對少見的系統性做法。

🔬 Layer 1 詳解：守恆律是物理模型的「地基」

Layer 1 是整個框架的地基。它驗證的不是某個特定模型的輸出精度，而是所有模型輸出都必須遵守的物理不可違反原則。以 Betz 限為例：無論風速、槳距角（pitch）或葉尖速比（λ）取什麼值，風能轉換係數 Cp 在物理上不能超過 0.593（Betz 定律）。若某組參數計算出 Cp = 0.7，這代表模型有嚴重錯誤，而不是「性能特別好的風機」。

類似的，能量守恆要求：P_aero × η_drivetrain × η_converter ≈ P_elec（容差 ±5%）。這個式子連接了氣動、機械傳動、電力轉換三個子系統，任何一個子系統的數值異常都會在這個約束中顯現。Layer 1 測試若失敗，代表整個模型鏈有根本性的錯誤，必須優先修復，後續層次的精細驗證才有意義。

三、負向測試：「故意改錯」的重要性

WMOM-23 的驗收條件中有一項特別值得注意：驗收含負向測試（negative test）——故意改錯物理常數，框架必須 catch 到錯誤。

這個要求背後有一個常見的測試陷阱：測試程式本身可能有 bug，導致即使模型是錯的，測試也會「誤放行」。負向測試強制要求開發者主動破壞已知正確的常數（例如把 Betz 係數設為 0.7），確認驗證框架確實會 fail——這等於是「驗證驗證器本身的有效性」。在安全攸關的工業應用中，這種 meta-validation 思維至關重要。

這讓人聯想到核電廠控制系統中的「trip test」傳統：定期故意觸發緊急停機程序，不是因為發生了緊急狀況，而是要確認緊急停機系統在真正需要時能夠正常運作。windMindOM 的負向測試哲學，是同樣思維在數位分身開發中的應用。

四、並行推進的物理深化項目

與 WMOM-23 同步進入 backlog 的，還有另外 5 個物理模型強化項目，形成一套完整的「物理正確性提升計畫」：

  • WMOM-24：修復 data quality 3 項 fail——個體差異 spread 異常與 Region 3 功率 CV 偏高，這是目前最影響模型可信度的已知問題。
  • WMOM-25：前端 RUL（剩餘使用壽命）+ 多頻段警報視覺化，讓物理計算結果能夠被清晰地呈現給運維人員。
  • WMOM-26：SCADA 標籤深度擴充——增加 25 個新標籤，涵蓋保護電驛狀態、冷卻迴路細部、變頻器內部數據與服務狀態。
  • WMOM-27：保護電驛協調模型（51/27/59/81），加入過電流、欠電壓、過電壓、頻率繼電器的協調邏輯，強化電氣側故障模擬。
  • WMOM-28：單齒 pitting/spalling 缺陷特徵模型，細化齒輪箱故障的振動 signature，提升對早期故障的識別精度。

🌊 離岸情境的物理驗證挑戰

相比陸上風電，離岸風場的物理模型驗證面臨更高的門檻。大氣邊界層特性在海面上與陸上顯著不同（低表面粗糙度、高TI、塔影效應增強）；機組長期浸泡在高鹽霧環境下，疲勞損傷累積的速率與陸上模型存在系統性差異；而颱風過境或極端海況的模擬，涉及超出 IEC 61400-1 標準設計工況的邊界條件。windMindOM 的物理驗證框架設計，需要特別考慮這些離岸特有的物理情境，未來論文研究的機會也在這裡。

五、從 Digital Twin 到「可信 Digital Twin」

windMindOM 的 WMOM-23 設計轉折，其實觸及了目前 digital twin 領域的一個核心問題：可信度（trustworthiness）如何被工程化地建立？

學術界對 digital twin 的研究多聚焦在「如何建立更精確的模型」（例如 Physics-Informed Neural Network、Bayesian filtering、Data-driven surrogate model），但對「如何持續驗證模型仍然是正確的」關注相對不足。隨著模型複雜度提升、多物理場耦合增加，一個沒有持續驗證機制的 digital twin，其可信度會隨著修改和演進而「悄悄衰減」——開發者無從得知模型在哪個 commit 之後開始偏離物理現實。

WMOM-23 的 7 層 Physics Validator + Layer 7 測試紀錄保存，提供的不只是「目前模型是否正確」的答案，更提供了「模型物理可信度隨時間演進的歷史記錄」。這個設計思想，可能成為 DOF Lab 未來發表關於「可信 Digital Twin 工程實踐」的重要研究素材。

🔬 DOF Lab 研究室行動項目

  • WMOM-23 Layer 1 開工：建立 tests/physics/reports/ 目錄結構與 YAML metadata 範本，接著實作 tests/physics/test_invariants.py，驗證 Betz 限與能量守恆；納入 1 個負向測試確認 fail 機制有效。
  • WMOM-24 同步進行：優先修復 3 項 data quality fail（spread + CV），為 Layer 1 通過提供乾淨的基線。
  • 研究方向探索：整理「Physics Self-Validation Framework for Digital Twins」的設計模式，評估是否有投稿 IEEE Transactions on Industrial Informatics 或 Renewable Energy 的潛力。
  • WMOM-25 前端規劃：待 Layer 1/2 完成後，將驗證結果的 health check report 整合進前端 dashboard，讓物理可信度狀態成為運維介面的一等指標。
  • parking lot 評估：完整 BEM 氣動負載、curled-wake yaw 偏斜模型、氣彈耦合 FEM 等進階物理已列入 parking lot，待 M5 之後再評估是否納入主線開發。

教授在 review 中的一句話，可以作為整個 WMOM-23 設計哲學的總結：「物理模型不能只是說有採用，要驗證結果是否準確，而且每次測試要留紀錄。」這不只是一個開發決策，更是一個關於研究嚴謹性的宣告——在追求功能完整的同時，絕不妥協基礎的物理正確性。這種取捨，正是讓工業 AI 研究從「可展示的 demo」走向「可信賴的工程系統」的關鍵一步。
