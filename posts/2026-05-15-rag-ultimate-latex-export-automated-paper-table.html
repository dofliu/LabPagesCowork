從實驗結果到論文表格：RAG Ultimate 的 LaTeX 自動化輸出系統

2026-05-15 • 劉瑞弘教授團隊 • RAG Ultimate / LaTeX / Research Automation / Academic Writing

RAG Ultimate LaTeX Automation Research Software Academic Writing TypeScript

研究者跑完一批實驗，拿到 precision、recall、faithfulness、latency 各項數字之後，最讓人頭痛的往往不是分析，而是把數字排進論文表格。對齊小數點、找出最佳值、加粗標記、計算統計顯著性、確認 booktabs 格式符合期刊規範……每一步手動操作都是潛在的錯誤來源，也是重複性極高的體力勞動。RAG Ultimate 今天完成了 PR #88 的合併，進一步強化了 latexExportUtils 模組的型別安全——這個模組正是為了把這套繁瑣工序自動化而生。

這篇文章介紹 RAG Ultimate 的 LaTeX 自動化輸出系統：它能做什麼、怎麼設計、以及今天的 PR 解決了哪個容易被忽視的隱性問題。

為什麼需要自動化 LaTeX 輸出？

在一般的研究工作流中，「從實驗到論文表格」通常是這樣的：在 Jupyter Notebook 或 Python 腳本裡產生一個 DataFrame，把它 print 出來，然後對著螢幕手動把數字貼進 .tex 檔。聽起來很直覺，但代價龐大：

  • 手動對齊：小數點位數不一致、單位不統一（ms vs. s），在 review 時被要求修改就要全部重來。
  • 最佳值標記遺漏：忘記把某個 metric 的最高值加粗，或是加粗了錯誤的數字，評審一眼就能看出來。
  • 實驗有更新時重做整張表：補跑了幾個 run、修正了某個 bug，整張表都要重製，而且不知道哪格有沒有同步更新。
  • 顯著性符號遺漏：統計顯著性（p < 0.05）標記要手動對照 t-test 結果，容易遺漏或標錯位置。

RAG Ultimate 的解法是把這一切封裝在 utils/latexExportUtils.ts 裡，讓研究者只需要傳入聚合好的實驗數據，就能拿到可以直接貼進論文的 booktabs 格式 LaTeX 程式碼。

latexExportUtils 的四個核心功能

generateComparisonTable()
多模式對比主表。輸入多個 RAG 模式的聚合結果，自動加粗最佳值、底線第二名，支援 ACL/EMNLP booktabs 格式（\toprule / \midrule / \bottomrule）。
generateSignificanceMatrix()
統計顯著性矩陣。依 p-value 輸入自動標注 †（p<0.05）、‡（p<0.01）等符號，生成各模式兩兩比較的顯著性表格。
generateAblationTable()
消融實驗表。針對某一基準模式，呈現逐步添加或移除元件後的效能變化，常見於論文的 Ablation Study 章節。
generateCrossFormatTable()
跨文件格式比較表。比較同一 RAG 模式在 PDF、Word、Markdown 等不同來源文件上的表現差異。

🔬 LaTeXTableOptions：精細控制每張表格

每個函式都接受 LaTeXTableOptions 參數，可控制：boldBest（是否加粗最佳值）、underlineSecond（是否底線第二名）、significanceMarkers（是否標注顯著性符號）、fontSize（表格字體大小，支援 footnotesize / scriptsize 以配合雙欄格式）、caption 與 label。論文投稿前只需調整這幾個旗標，不需改動核心邏輯。

PR #88：看似小修，實則關鍵的型別安全升級

今天合併的 PR #88 修改的是測試檔案 services/__tests__/latexExportUtils.test.ts，乍看只是「把字串改成 enum」，但背後邏輯值得深入說明。

修改前，測試 mock data 長這樣：

// 修改前：以字串字面值繞過型別檢查 { mode: 'Basic RAG' as any, chunkingStrategy: 'Recursive Character' as any, // ... }

修改後：

// 修改後：使用 enum member，拿回型別安全 { mode: ModeType.BASIC_RAG, chunkingStrategy: ChunkingStrategy.RECURSIVE, // ... }

⚠️ as any 的隱性風險：改了 enum 卻不知道測試已失準

假設三個月後，因應新的 RAG 模式命名規範，ModeType.BASIC_RAG 的底層字串值從 'Basic RAG' 改成 'basic-rag'。使用 as any 的測試 mock 不會跟著更新——TypeScript 編譯器不會報錯，jest 可能也不會失敗（如果測試沒有直接比較字串值的話）。結果是：測試繼續通過，但測試的對象已經不再反映真實程式的行為。使用 ModeType.BASIC_RAG 後，未來的重新命名會立刻在編譯期被抓到，不會靜默失準。

這個道理在研究程式碼中尤其重要。LaTeX 輸出函式所顯示的欄位標籤，直接對應到 ModeType 和 ChunkingStrategy 的字串表示。一旦這些字串在測試中失準，就可能出現「測試全過，但生成的表格欄位名稱和論文本文描述對不上」的情況——這種錯誤在 peer review 的截止日前最容易出現，也最難追蹤。

研究可重現性的基石：從資料到論文的型別安全鏈

RAG Ultimate 的設計哲學之一，是讓「從原始實驗資料到論文截圖」這條路盡可能自動化且可驗證。這包含幾個層次：

  • 資料層：RAGAS 評測結果存入 Qdrant 向量資料庫，每筆記錄帶有實驗 run ID、模式、切塊策略等結構化欄位。
  • 聚合層：academicExportUtils.ts 負責從多個 run 聚合均值與標準差，產生 AggregatedResult[]。
  • 輸出層：latexExportUtils.ts 將聚合結果轉換為 booktabs LaTeX，直接貼進 .tex 主檔。
  • 驗證層：2,524 個測試（含 PR #88 後的 latexExportUtils tests）確保每一個轉換步驟的正確性，TypeScript 嚴格模式保障型別一致。

當審稿人要求「補充 95% confidence interval」或「加入 cross-document-format 比較」，研究者只需在聚合層補上計算邏輯，呼叫對應的 generateXxxTable() 函式，即可立刻拿到更新後的 LaTeX——而不是重新開始一輪手動作業。

💡 對學生的啟示：測試 mock 也是論文的一部分

許多研究生把測試 mock data 視為「隨便填就好的假資料」。但在研究軟體中，測試 mock 實際上定義了你對系統行為的斷言——它說的是「當輸入是 ModeType.BASIC_RAG 時，輸出的 LaTeX 應該包含 Basic RAG 這個欄位名稱」。如果 mock 的型別是 any，這個斷言就有一個默默漏水的缺口。把 mock 改成強型別的 enum，是讓測試真正說出你的意圖。

下一步：Stage 2B 與論文主表的完整 run

PR #88 完成後，RAG Ultimate 的 latexExportUtils 測試基礎已經扎實。下一個里程碑是 Stage 2B——ResearchLab 的雙欄重組（Experiment Plan / Live Telemetry），以及補跑 r06/r07 讓完整 run 數達到 ≥4，為論文主表提供統計上足夠的重複次數。一旦四輪完整實驗資料就位，generateComparisonTable() 就能自動生成論文的核心比較表，配合 generateSignificanceMatrix() 呈現各模式間的統計顯著性，整個投稿流程將大幅縮短。
