windMindOM M2 里程碑：把 ECN Cost Engine 移植進 Python 與離岸風場運維 SaaS 的下一步

2026-05-05 • 劉瑞弘教授團隊 • windMindOM / ECN Tool / 離岸風場運維 / O&M SaaS

windMindOM ECN Cost Engine 離岸風場 O&M Work Order

2026 年是台灣離岸風電的關鍵轉折點。第三階段（Phase 3）風場陸續商轉，Cadeler / Nexra 等國際 O&M 服務商已在 2026 年初簽下超過 €2,000 萬歐元的台灣運維合約。然而，現場工程師的日常工具，仍然是 Excel 試算表、LINE 群組，和一本本紙本手冊。這個落差——從十億規模的風場資產到工程師用手機截圖傳派工指令的現實——正是我們開發 windMindOM 的初心。

本週，windMindOM 迎來了一個值得記錄的里程碑：M2 Cost Module 在一天之內完工，比預定的六月完工時程提前了整整一個月。與此同時，M3 Work Order 模組的設計階段正式啟動。這篇文章記錄這個過程，並談談「把工業標準的 ECN O&M Cost Tool 移植進 Python」這件事對整個產品的意義。

ECN O&M Cost Tool：業界標準，但只剩 Excel

ECN（荷蘭能源研究中心，現已併入 TNO）發布的 O&M Cost Estimator（OMCE-Calculator）是目前離岸風場成本分析的業界黃金標準。這套工具採用時域模擬（time-domain simulation）方式，對三類維護事件——非計畫性糾正維護（unplanned corrective maintenance）、狀態監測維護（condition-based maintenance）、預防性維護（preventive maintenance）——分別建模，輸出整個風場生命週期的 LCOE（度電成本）與成本拆解。

問題是：ECN Tool 5.0 的實作是 Excel + VBA。對台灣運維廠商而言，這意味著每月要交給業主的 O&M 成本報表，仍然需要手動填表、複製貼上、再找懂 ECN 的顧問核對——整個流程既費時又容易出錯，且無法與 SCADA 監控數據自動串接。

💡 windMindOM 的核心差異化：ECN 計算 → Python API

windMindOM 的 Cost Module（modules/cost/）將 ECN 的核心計算邏輯移植為 Python 引擎，包含四個子模組：

  • cost_cal：主要成本計算引擎（材料、工時、吊掛費用）
  • waiting_time：等船時間與天氣視窗建模（離岸風機特有的後勤瓶頸）
  • monte_carlo：蒙地卡羅不確定性分析（N 次模擬取信賴區間）
  • var_fluct：變動成本波動估算（燃油、匯率、工資調漲）

加上 Cost Adapter 負責 windMindOM 內部 schema ⇄ ECN 計算 schema 的轉換，以及 POST /api/cost/forecast、GET /api/cost/ledger 兩個 REST API endpoint，讓成本計算從「填 Excel」變成「呼叫 API」。

M2 在一天內完工：這代表什麼？

M2 Cost Module 原本預計是六月的工作，結果在一個工作天內完成了全部 8 個 sub-task（skeleton 架設 → 四個 engine 移植 → adapter → schemas → API → frontend）。這並不是因為「切了很多角落」，而是有幾個關鍵原因：

  • ECN 邏輯已在 ECN_tool 專案高度模組化：之前在 researchTopic/ECN 做的計算原型，已將四個 engine 分離，移植時幾乎是「複製 → 整理 imports → 加 FastAPI router」。
  • K13 demo dataset 快速驗證：使用北海 K13 風場 demo data 立即跑通，計算結果與 ECN Tool 5.0 Excel 版本對齊，確認移植正確。
  • Schema-first 設計：先定義 Pydantic schemas，讓 API 與 frontend 開發可以並行，大幅減少整合等待時間。

提前完成 M2 的另一個意義在於：這驗證了 windMindOM v0.8.1 的核心假設——以 digiWindTurbine（我們的 SCADA 模擬器平台，完成度約 90%）為基礎，再依序疊加功能模組，確實比從零開始架 platform 快得多。

M3 啟動：Work Order 生命週期設計

M2 完工後，M3 Workflow Part 1（Work Order + Approval）隨即進入設計階段。目前已完成的是 DN-01 Work Order Lifecycle 雛形，定義了一張派工單從建立、審核、派遣、執行到結案的完整狀態機。

為什麼 Work Order 是高優先度的模組？因為運維廠商最痛的日常場景是：「現場工程師用 LINE 說『2號機齒輪箱異音，需要換零件』→ 辦公室主管用 LINE 回『好，我叫人過去』→ 到底派誰、帶什麼料、費用怎麼算，全在各自的腦袋或 Excel 裡。」

windMindOM 的 Work Order 模組要解決的不是「軟體功能」的問題，而是讓現場工程師在 30 秒內完成派工建立、讓主管在手機上核准、讓成本自動更新進 Cost Module。這三個動作打通，才能取代 LINE + 紙單的組合。

現況（運維廠商） windMindOM 目標
LINE 群組傳派工指令 Work Order API + mobile UI
Excel 手動記錄成本 Cost Module 自動匯總
月底手找 SCADA 數據報給業主 SCADA dashboard + 自動月報
故障查手冊靠 LINE 問師傅 RAG 知識庫（M5）
成本估算找顧問跑 ECN Excel POST /api/cost/forecast（已完成）

接下來 M3 的開發路線是：WMOM-15（Work Order domain walkthrough）→ WMOM-16/17（domain model + CRUD API）→ WMOM-18（Approval 狀態機）→ WMOM-19/20（前端介面）。目標在七月完成第一個可 demo 的 Work Order 流程。

▶ 研究室行動項目

  • 技術：WMOM-15 開始 Work Order domain walkthrough，與 cost module 的 cost↔farm 整合（WMOM-10）對齊
  • 驗證：與 1-2 個 friendly 運維廠商進行 M1 demo，確認「simulator 產生的 SCADA + Cost dashboard」是否能打動客戶
  • 論文：windMindOM 的 ECN Python 移植 + 蒙地卡羅不確定性分析，可作為「離岸風場 O&M 決策支援系統」期刊論文素材
  • 課程：ECN Cost Engine + SCADA 模擬器的教學案例，適合「智慧電廠管理」相關課程的期末專題

台灣的離岸風電 2026 到 2035 年預計新增 15 GW 裝機容量，以每年 1.5 GW 的速度推進。隨著第三階段風場進入商轉，O&M 服務需求將呈指數成長，而現場人力與成本控管的壓力也隨之上升。windMindOM 的目標，是在這波浪潮中，成為台灣運維廠商最實用的數位工具——不是因為它最「AI」，而是因為它能取代現場工程師手邊的 LINE + Excel + 紙單，讓每一筆派工、每一項成本，都有可追溯的數位紀錄。
