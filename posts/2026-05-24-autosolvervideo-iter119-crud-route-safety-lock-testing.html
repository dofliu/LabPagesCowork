安全鎖測試哲學：AI 輔助迭代開發中，補上 Route-Level 覆蓋缺口的系統思維

2026-05-24 • 劉瑞弘教授團隊 • Testing / FastAPI / AI-Assisted Development / Research Engineering

autoSolverVideo iter 119 Route-Level Testing FastAPI / pytest AI 輔助開發

在 AI 輔助開發的高速迭代節奏下，一個現象幾乎必然發生：功能已上線，但對應的測試從來沒被補上。 不是沒打算補，而是當時交付壓力下，功能通了就合併了；測試計畫留在心裡，最後靜靜地消失在 backlog 裡。 這種狀態，在 DOF Lab 的 autoSolverVideo 教學影片自動生成平台，被稱為「裸奔路由」—— API endpoint 在 production 上跑著，但任何重構都是在沒有測試網的空中鋼索上行走。

iter 119 做的事情很單純，卻很有代表性：為 jobs.py 四個核心 CRUD endpoint 補上完整的 route-level 測試覆蓋。 23 個新測試、0 行 production code 改動、測試計數從 1,397 增加到 1,420。 這不是功能迭代，而是一次「還債」——把技術欠下的測試債，有條不紊地還清。

一、為什麼是 CRUD 優先？

autoSolverVideo 的後端是典型的 FastAPI 架構，核心流程是： 使用者提交一個「工作」（Job）——可能是考題 PDF、教學簡報、或是網路文章連結—— 系統透過 Gemini 處理內容、產生腳本、合成語音、輸出 MP4，最後自動上傳 YouTube。 這條流水線的起點，就是 POST /jobs 這個建立工作的 endpoint。

然而，server/routes/jobs.py 裡的四個 CRUD 路由—— POST /jobs（建立）、GET /jobs（列舉）、 GET /jobs/{id}（查詢單筆）、DELETE /jobs/{id}（刪除）—— 自從 PR-3a 合併上線後，就一直處於「裸奔」狀態。 iter 117 補了 draft/log/section_render/artifacts 五個 endpoint，iter 118 補了 proposals 的 duplicate， 但 CRUD 本身始終是一個懸而未決的缺口。

⚠️ 裸奔路由的隱形風險

在 AI 輔助的高速迭代環境中，每一次 refactor 都是潛在的地雷。 source_type 的早期驗證分支（URL scheme 白名單、path 存在檢查、缺欄位攔截）、 created_at 的降序排序邏輯、cache 與 disk 的同步 cleanup—— 這些邏輯只要被無意間改動，就會直接上線，沒有任何測試攔截。 不是因為粗心，而是因為沒有安全網。

二、安全鎖的設計哲學

iter 111 起，autoSolverVideo 的測試策略確立了「route 安全鎖」的思維： 對每一個 endpoint，先鎖住它的外部契約（輸入驗證、輸出格式、HTTP 狀態碼、副作用）， 讓任何偏離契約的 refactor 立刻被 CI 攔截。 這不是傳統意義上的 TDD——測試是在功能上線後才補的——但補測試的過程本身就是一次「重新思考設計」的機會。

測試類別 測試數 鎖住的契約重點
TestCreateJobEndpoint NEW 11 全 source_type 建立路徑、exam_pdf 的 require_review 硬規則（強制 True）、URL scheme 白名單（http/https 接受、ftp 400）、缺欄位與不存在 path 攔截、schedule_job 觸發驗證
TestListJobsEndpoint NEW 4 空列表回傳、單工作命中、3 工作降序排序（顯式 created_at 避免 Windows microsecond 退化）、跨 source_type 混合列舉
TestGetJobEndpoint NEW 2 存在工作的 200 回應與全欄位驗證（id/state/source_type/timestamps/youtube_uploads）、不存在 ID 的 404
TestDeleteJobEndpoint NEW 6 204 空 body、不存在 404、刪後 GET 404（cache 清乾淨）、刪後不出現在 list、disk dir rmtree、double delete 第二次 404

值得特別記錄的一個細節：TestListJobsEndpoint 的排序測試，刻意為三個工作設定了 顯式且相差一分鐘的 created_at 時間戳，而非依賴系統自動生成。 原因是 Windows 環境下 datetime microsecond 精度退化的邊角案例—— 在某些環境下兩筆記錄的時間戳「看起來相同」，排序就會變成不穩定的 stable order， 導致 CI 上通過、本機失敗的詭異現象。 這個細節看似微小，卻是跨平台測試可靠性的重要體現。

🔒 安全鎖的核心價值

「安全鎖」不是為了讓測試覆蓋率數字好看，而是為了讓每一次 AI 輔助的快速 refactor 都有一道自動化的防線。當 Claude 幫你重構 source_type 驗證邏輯時，23 個 CRUD 測試會在背景靜靜地確認：外部行為沒有改變。 這就是「讓 AI 開發更安全」的工程實踐。

三、同步進行：人臉辨識點名系統的品質防護

同一天，人臉辨識點名系統（Face Recognition Attendance）也有值得記錄的進展。 系統在人臉品質評分模組新增了針對 blur 軸（模糊程度）的跨層單調性回歸測試—— 固定其他變因（face bbox、sim/gap/z），只改變影像模糊程度，驗證品質分數是否呈現預期的單調遞減趨勢。 如果未來有人重構評分函式，這個測試會立刻告訴你：「你改壞了 blur 軸的物理直覺。」

此外，系統也實作了「人工覆核動態同步」機制——監考過程中的嚴重度計數現在支援對未知類別的防禦性靜默跳過（silent-skip guard）， 確保即使遇到訓練時未見過的異常標籤，系統也不會拋出例外、中斷服務。 這是一個典型的「防禦性程式設計」實踐，在教學現場尤其重要：課堂不等人，系統必須穩定。

💡 研究室今日觀察

今天兩個系統的更新，都沒有新增任何功能。但它們讓系統變得更值得信任。 在 AI 輔助開發的世界裡，速度是我們最大的優勢，但也是最大的風險來源。 測試、防禦性設計、以及對「裸奔路由」的系統性盤點，是讓這個速度可持續的工程基礎。 這種思維，我們稱之為「穩健的快」（robust speed）——不是慢下來，而是讓每一步都站得住腳。
