windMindOM M4 完工：從離岸風場工單流程到 RAG 知識整合的下一哩路

2026-05-25 • 劉瑞弘教授團隊 • Wind Energy / O&M SaaS / RAG / Product Engineering

windMindOM M4 完工 離岸風能 O&M RAG 知識系統 FastAPI / React / SQLite

每一個研究平台都有一個關鍵的轉折點——從「能跑起來」到「真正可以交給人用」。 DOF Lab 的 windMindOM（風心智運維平台）在 2026 年 5 月 10 日迎來了這個轉折： M4 里程碑端到端貫通，劉瑞弘教授親自實測完整工單生命週期， 從建單 → 派工 → 開始 → 進度紀錄 → 完工 → 員工簽名 → 組長審核（通過/駁回）→ 關單， 全流程順暢通過。

這不只是一個技術檢查點的達成，而是 windMindOM 作為一個離岸風場運維 SaaS 工具 走向第一個客戶 PoC 的重要里程碑。本文記錄 M4 的完工樣貌、這次衝刺中踩過的 四連環 bug，以及接下來 M5 知識系統整合的規劃思路。

一、M4 的核心成就：工單生命週期端到端貫通

windMindOM 的 M4 里程碑聚焦在工作流程第二部分（Inventory + Reporting）， 延續 M3 建立的工單核心（Work Order + Approval），加入了報表生成後端、月報 PDF 下載， 以及一系列讓整個平台「真正能用」的細節修補。

M4 衝刺最具代表性的一天，是一場典型的「一日四 PR + 一個 cherry-pick + 一輪 bug 修復」馬拉松。 當天完成了 A8 報表後端（PR #23）和 A9 報表前端（PR #24），看似已接近收尾， 但緊接著劉老師的實際測試踩到四連環 bug：

  • CORS spec violation：跨來源請求設定不符 RFC，在特定瀏覽器版本觸發
  • work_orders 資料表名稱衝突：與既有表名撞名，導致 migration 失敗
  • dispatch guard 邏輯漏洞：派工防護條件判斷有邊角案例未覆蓋
  • weather_window UX 體驗問題：天氣視窗欄位在特定工單狀態下顯示邏輯不對

關鍵在於：每一個 bug 都修到了根本原因（root cause），沒有表面 patch。 CORS 和資料表衝突合併成 hotfix PR #25，dispatch 和 assignee 問題合併成 PR #26， weather UX 則透過 cherry-pick 精準接入（commit 9c8f476）。 這種「一次發現、徹底根治」的修復哲學，正是研究平台走向商業產品所需要的工程紀律。

🏆 M4 最終里程碑驗證結果

後端 API 零迴歸（502 → 504 全通過），前端 TypeScript 編譯零錯誤（tsc 0）， Vite 建置時間 6.06 秒、733 個模組。所有 PR 分支已合併並刪除（local + remote）， main 分支處於 100% production-ready M4 狀態。 月報 PDF 下載功能正常，工單完整生命週期：建立 → 派工 → 執行 → 簽名 → 審核 → 關單全部貫通。

二、六個里程碑的產品藍圖

windMindOM 的整體設計從一開始就是為了商業落地，不是純學術平台。 六個里程碑的設計充分反映了這個目標：

M1 ✓ 基礎設施建立（WMOM 架構、Docker Compose、Modbus TCP、OPC UA Bachmann Z72 連線） 2026-05
M2 ✓ 成本模組（從 ECN 移植：成本引擎 × 4 + adapter + schemas + API + 前端，一天完成） 2026-06
M3 ✓ 工單系統 Part 1（Work Order + Approval：後端 5 PR + 前端 2 PR 全部完成） 2026-07
M4 → 工單系統 Part 2 + 報表（Inventory 前端 A6/A7 + E2E 測試 A10 + follow-up F1-F6 待完成） 2026-08
M5 知識系統 + RAG 整合、現場行動裝置 UI（與 RAG_Ultimate 平台對接） 2026-09
M6 第一個離岸風場運維廠商 PoC + 第一筆合約（目標 NT$2–4M） 2026-10

值得注意的是 M2 的完成速度：成本模組的四個引擎、adapter、schemas、API 端點和前端頁面， 在一天內全部完成。這不是靠蠻幹，而是因為有來自 ECN 工具的紮實基礎可以移植， 以及 AI 輔助開發讓程式碼產生與整合的節奏大幅壓縮。 windMindOM 的核心資產之一，就是從 digiWindTurbine 繼承的 104 個 SCADA 標籤、11 個故障場景物理模型，以及完整的大氣穩定度耦合模型—— 這些「紮實的底盤」讓每一個上層功能都能快速落地。

三、M5 的核心挑戰：讓現場人員能「問」平台

M4 完工後，windMindOM 擁有了完整的工單流程、成本計算、和報表輸出能力。 但離岸風場的運維工作還有一個關鍵需求，是現有平台還無法滿足的： 現場技術人員需要能快速查詢設備維修歷史、故障排除手冊、和法規合規要求。

這正是 M5「知識系統 + RAG 整合」的出發點。在 DOF Lab 的技術棧裡， 已經有一個成熟的企業級 RAG 平台——RAG_Ultimate——支援 17 種檢索模式、 Qdrant 向量資料庫、多文件知識庫管理，以及完整的 RAGAS 評測實驗室。 M5 的核心任務，就是在 windMindOM 的現場 UI 中接入 RAG_Ultimate 的知識查詢能力。

💡 為什麼離岸風場特別需要 RAG？

離岸風場的維護文件量龐大：IEC 61400 系列技術標準、Bachmann Z72 PLC 操作手冊、 各風機型號的維修程序書、歷史事件日誌、備料採購規格…… 傳統關鍵字搜尋在這種異質文件環境下效果很差。 RAG（Retrieval-Augmented Generation）能夠跨文件理解語意， 讓現場人員用自然語言提問：「主齒輪箱更換後的首次維護週期是多少？」 並直接得到有來源依據的回答，而不是一頁頁翻 PDF。

DOF Lab 目前已有「Z72 風機運維專家系統」的 14 個情境案例（發布於 2026-05-19 的研究日誌），為 windMindOM M5 提供了知識庫的第一批種子文件。 接下來的整合路徑，將從最高頻的「故障排除查詢」場景出發， 逐步擴展到「預防性維護建議」和「備料採購決策支援」。

四、從研究平台到商業 SaaS 的工程心法

windMindOM 的開發歷程揭示了一個從研究平台走向商業產品的重要心法： 「現場實測 → 踩 bug → 根治 → 再測」的循環，是品質的唯一保證。

M4 衝刺中最大的教訓，在 STATUS.yaml 裡寫得很清楚： 「今日 hotfix 風暴源於沒有 lifecycle E2E test——強烈建議下次 session 第一件事做 A10。」 這個教訓很典型：在高速迭代的研究環境中，功能完成了就往前衝， 端到端的整合測試往往是最後才補的。但在一個真正要交給運維廠商使用的平台， 沒有 E2E test coverage 的上線，就像在海上作業沒繫安全繩。

⚠️ M4 尾款：A10 E2E 測試是 M5 前的優先項

A6/A7 領料庫存前端、A10 E2E 測試、以及 F1–F6 的各項 follow-up， 是 windMindOM 進入 M5 之前必須完成的工作。其中 A10 E2E 測試被劉老師明確列為 「下次 session 第一件事」，因為一個沒有端到端測試覆蓋的工單系統， 在客戶 demo 環境中隨時可能因為微小改動而崩潰——這是商業 PoC 絕對無法接受的。

windMindOM 繼承自 digiWindTurbine 的豐富物理模型基礎（Bastankhah-Porté-Agel 高斯尾流、 大氣穩定度 × 剪切 / TI / 尾流 / 偏向五重耦合、疲勞壽命 DEL 計算）， 讓它在技術深度上超越了大多數同類工具。M5 加入 RAG 知識系統後， 這個平台將同時具備「物理感知」（數據驅動的設備監控）和「文件感知」（語意搜尋的知識查詢）兩個維度， 成為一個真正接近全棧的離岸風場運維工具。

🎯 DOF Lab 接下來的行動

  • 短期（M4 尾款）：完成 A10 E2E 測試，確保工單系統在 demo 環境的穩定性；完成 A6/A7 領料庫存前端
  • 中期（M5 規劃）：定義 windMindOM ↔ RAG_Ultimate 的 API 介面規範；建立 Z72 風機知識庫第一版（14 個情境 → 完整手冊）
  • 現場 UI 設計：開始 M5 行動裝置介面的 UX 研究，針對現場作業人員的使用情境（手套、強光、單手操作）設計互動流程
  • 商業準備：準備第一個潛在客戶（離岸風場運維廠商）的 PoC 展示腳本，確認目標客戶的實際痛點清單

離岸風能的運維管理，長期以來是一個依賴經驗傳承和紙本手冊的產業。 windMindOM 的願景，是用 AI 加持的數位工具，讓台灣的離岸風場運維廠商能夠以更低的成本、 更高的效率管理設備健康和維修流程。M4 的完工，讓這個願景從「研究原型」 邁向了「可以讓人試用的產品」。而 M5 的知識系統整合，將是讓它從「好用的工具」 成長為「不可替代的平台」的關鍵一步。
