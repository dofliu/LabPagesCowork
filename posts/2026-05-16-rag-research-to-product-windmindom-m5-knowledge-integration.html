研究成果如何轉化為產品功能：RAG_Ultimate 與 windMindOM M5 的解耦整合設計

2026-05-16 • 劉瑞弘教授團隊 • RAG / windMindOM / System Architecture / Research-to-Product

RAG_Ultimate windMindOM System Design Research-to-Product Offshore Wind O&M

研究室同時維護兩個大型系統：RAG_Ultimate（企業級 RAG 研究平台，17 種檢索模式、829 支測試、Qdrant 向量資料庫）與 windMindOM（離岸風場運維廠商 SaaS 產品，5 大功能模組）。這兩個系統在技術棧上高度相關，卻又必須保持獨立——一個是研究平台，一個是要交付給真實客戶的產品。

隨著 windMindOM M4（庫存派工 + 報表）在本月初正式完成端到端測試，M5「Knowledge / RAG + 現場 mobile UI」已正式排入 2026-09 的里程碑。如何讓研究平台的成果真正流入產品，同時又不讓兩個系統彼此綁架，是我們在規劃 M5 時遇到的核心設計問題。這篇文章記錄我們的解答。

問題根源：研究環境與產品環境的天然矛盾

在研究室裡，RAG_Ultimate 有完整的開發環境：Qdrant 容器、FastAPI 後端、React 前端評測介面。我們可以隨時切換 embedding model、調整 chunking 策略、對比 RAGAS 指標。這種靈活性是研究工作的命脈。

但 windMindOM 的客戶——離岸風場的運維廠商——有截然不同的需求。他們需要一個可以在風場現場部署的工具，不能依賴連線到研究室的 Qdrant 實例，不能在客戶端安裝 Python 研究環境，更不能每次研究員調整實驗參數就讓客戶系統崩潰。

🎯 核心矛盾

研究要求靈活可調（探索未知），產品要求穩定可部署（服務真實客戶）。若直接讓 windMindOM 在 runtime 呼叫 RAG_Ultimate 的 API，任何一方的更新都可能破壞另一方——這是一個必須在架構層面解決的問題。

解耦的核心思路：Artifact 交付而非 Runtime 整合

我們的設計決策非常明確：RAG_Ultimate 不作為 windMindOM 的 runtime 依賴，而是作為研究工件（artifact）的生產者。具體而言，RAG_Ultimate 在研究完成後交付兩類檔案：

  • 策略檔（rag_strategy_z72_manual.yaml）：描述該機型技術手冊的最佳 RAG 策略，包含 chunking 方式、embedding model 選擇、檢索模式組合等超參數。
  • 預計算向量檔（.qdrant_snapshot）：已對特定文件集（如 Z72 機型操作手冊、警報碼說明書）建立好的 Qdrant collection snapshot。

windMindOM M5 的 knowledge/ 模組只負責載入這兩類檔案，執行查詢，並將結果渲染到前端介面。整個資料流如下：

→

這個架構讓兩個系統各自獨立演進：研究平台可以任意調整實驗設計，只要最終交付的 artifact 格式保持相容；產品端則不需要了解 RAG 的內部機制，只需讀取策略檔並載入向量快照即可。

💡 為什麼這個設計在工業 AI 場景格外重要

離岸風場的 IT 環境通常有嚴格的網路隔離——控制室不一定能連到外部雲端。「預計算向量檔 + 本地 Qdrant 容器」的模式讓知識檢索可以在完全離線的環境中運作，這對於真實的工業部署場景是關鍵優勢。

現場工程師的使用情境：警報頁面的 RAG 整合

M5 的核心使用情境是 /field/alerts/{id} 頁面。當風機觸發警報，現場工程師開啟這個頁面時，畫面左側顯示即時 SCADA 數據與警報詳情，右側即時呈現 RAG 查詢結果——從技術手冊與歷史工單中撈取與該警報最相關的處置步驟。

這個設計的關鍵不是「AI 有多聰明」，而是「工程師在現場最需要什麼」。不需要生成式 AI 創造新知識——現場工程師需要的是快速找到機型手冊裡那一頁關鍵的處置說明。RAG 在這裡做的是精準索引，而非創作。

📐 M5 知識模組的技術邊界

windMindOM 的 knowledge/ 模組預計包含三個核心檔案：
strategy_loader.py：讀取 RAG_Ultimate 交付的策略檔
vector_store.py：管理本地 Qdrant snapshot 的載入與查詢
knowledge_router.py：FastAPI router，提供 /api/knowledge/query endpoint

有意設計的邊界：M5 不自行處理 chunking、不訓練 embedding model、不調整 RAG 超參數——這些全部由 RAG_Ultimate 在研究環境中處理完畢後以 artifact 形式交付。

更新策略：不停機升級知識庫

當 RAG_Ultimate 研究出更好的 chunking 策略或切換到更強的 embedding model 時，如何讓 windMindOM 的客戶無縫升級？我們的設計是分版本的向量快照交付機制：

  • 小幅更新（策略調整）：只更新 YAML 策略檔，保留現有向量快照。windMindOM 讀取新策略後改變查詢行為，不需要重新建索引。
  • 中幅更新（新文件加入）：RAG_Ultimate 重切新文件並增量加入 collection，交付 diff snapshot。windMindOM 合併後繼續提供服務。
  • 大幅更新（換 embedding model）：RAG_Ultimate 重新建立完整 collection，交付全量 snapshot。windMindOM 在維護視窗期間替換，舊版保留備份以支援回滾。

這套分級更新策略讓我們可以在不中斷客戶服務的前提下，持續將研究進展注入產品。對運維廠商而言，他們不需要了解 RAG 的內部機理，只需要在更新提示出現時下載新的知識包並讓系統自動載入即可。

🔬 研究室行動項目（M5 前置準備）

  • RAG_Ultimate 方面：完成風電技術文件的 benchmark 實驗（Z72 機型手冊 + 警報碼說明書），確定最佳 chunking 策略（目前候選：Semantic Chunking vs. Fixed 512 token + 20% overlap）
  • RAG_Ultimate 方面：設計 artifact 打包格式，確保 qdrant snapshot + strategy YAML 的版本相容性保證
  • windMindOM 方面：M5 開工前，先完成 A10 E2E 測試（M4 遺留項目），確保 regression test coverage 足夠
  • windMindOM 方面：設計 knowledge/ 模組的 API contract，讓前端可提前開始 mock 開發
  • 論文方面：RAG_Ultimate 的風電文件 benchmark 結果可直接成為期刊論文的實驗數據，M5 的工程實踐案例可作為應用段落

小結：「研究平台作為工件生產者」的設計哲學

RAG_Ultimate 與 windMindOM 的整合方式，體現了一個更廣泛的設計哲學：研究系統的產出不應該是 API，而應該是可獨立部署的工件。這讓研究和產品團隊各自保持最高的自主性，同時又能讓研究成果真正流入能服務真實用戶的產品。

對於同樣面臨「研究平台 vs 產品化」抉擇的學術團隊，這個模式值得參考。它的代價是需要明確定義 artifact 格式的 schema，並在兩個系統之間維護版本相容性契約；但帶來的好處是兩個系統都能以自己最適合的節奏演進，不會因為耦合而相互拖慢。

windMindOM M5 預計在 2026 年 9 月完成，屆時 DOF Lab 將有一個完整的「研究 → 工件 → 產品」管線的實際案例，期待屆時帶給大家更完整的心得分享。
