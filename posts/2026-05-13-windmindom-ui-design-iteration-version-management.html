工業介面的版本考古學：windMindOM 如何管理 4,600 行設計史料與啟動 v2 改版

2026-05-13 • 劉瑞弘教授團隊 • windMindOM / UI Design / Design Systems / Industrial B2B Software

windMindOM UI Design Design Systems 版本管理 工業 B2B

2026 年 5 月 12 日深夜 11 點，windMindOM 的 Git 記錄出現了一筆不尋常的 commit。它不是功能修復，也不是測試補充，而是一個 chore：把 14 個設計檔案、4,600 多行的 React 元件與介面規範，從 repo 根目錄搬進 docs/design/2026-05-07-ui-source/，並在 ISSUES.md 開了一個空白的 placeholder——WMOM-20260513-01「UI 改版 v2」，等待劉老師補上新的設計交接書才開工。

這個動作看似瑣碎，卻體現了一種在工業 B2B 軟體開發中常被忽略的重要實踐：設計史料的系統性保存。在功能衝刺的壓力下，設計決策往往隨著版本迭代而消失在 git history 的深處；而當下一輪改版需要回溯「當初為什麼這樣設計」時，答案往往已不可考。windMindOM 選擇了不同的路。

一、Calm Operator：一個設計理念的誕生與完成

時間回到 2026 年 5 月 7 日。windMindOM M3 工單系統剛完成，M4 的庫存派工正在推進。此時，劉老師提出了第一輪正式 UI 設計改版需求：平台的操作介面需要能在離岸風場這種高壓、高噪訊的工作環境中，讓技術員保持清醒的操作判斷。這個需求催生了「Calm Operator」設計理念。

Calm Operator 的核心哲學借鑒了工業控制室（Control Room）的設計原則：深色背景降低視覺疲勞、關鍵狀態用顏色而非文字快速傳達、密集資訊透過層級分組而非平鋪展示。設計團隊（與 Claude Opus 協作）在一個 session 內產出了三組 App 變體（VA、VB、VC）和三組頁面變體（V1、V2、V3），共同探索在同一個設計語言下的不同表達方式。

🎨 設計變體 VA / VB / VC

三套整體設計方向探索——在配色、排版密度、資訊層級組織上的不同取捨，各自有完整的 React 元件實作。

📐 頁面變體 V1 / V2 / V3

針對同一頁面的三種佈局策略——從寬鬆的儀表板風格到緊湊的工程師工具風格，提供具體的選擇對照。

🧩 Shared 元件庫

MiniDashboard、i18n 模組等共用元件，確保三套變體在資料層與互動邏輯上的一致性，便於 A/B 比較。

M4 完工後，Calm Operator 設計已完整體現在現行介面中：工單列表、派工流程、月報 PDF 下載，每一個頁面都沿用同一套設計語言。設計的使命完成了，但設計的記憶必須留存。

二、為什麼要把設計史料當作程式碼一樣版本管理？

多數開發團隊對「設計稿」的態度是：用完即丟，或放在 Figma 等工具中緩慢腐爛。這在消費級產品中或許可以接受，因為使用者對介面的記憶短暫，改版代價相對低。但在工業 B2B 軟體中，情況截然不同。

離岸風場的運維人員在與平台打交道時，有極高的肌肉記憶依賴——他們在海上作業時，注意力必須分配給環境安全與設備狀態，不允許因介面改版而重新適應操作流程。因此，windMindOM 的任何 UI 改版都必須有充分的設計依據：為什麼改？哪些元素要動？哪些必須保持不變？

💡 設計考古學的核心問題

當下一輪 UI 改版團隊問「當初為什麼把派工按鈕放在這裡？」或「為什麼選擇深藍而非深灰作為主背景？」時，他們應該能從 docs/design/2026-05-07-ui-source/WMOM 介面改版交接書.md 找到完整的設計決策記錄，而不是從渺茫的 Git commit message 中拼湊答案。

把設計檔案放入 Git 版本控制，並附上清楚的設計交接書（Design Handoff Document），是 windMindOM 在這方面的選擇。這讓設計不再是「有人記得的口耳相傳」，而是可以被搜尋、引用、對照的工程知識。

此外，當 AI 協作成為設計流程的常態，這個實踐變得更加重要。Claude Opus 在下一輪 UI 改版 session 中，可以直接讀取 VA.jsx、VC.jsx、design-canvas.jsx 這 48KB 的完整設計畫布，理解 v1 的設計語言後再提出 v2 的改進方向，而不必從零開始描述整個設計系統。這是 LLM 輔助設計的新工作模式：把設計史料當作 context，而非憑空生成。

三、WMOM-20260513-01：下一輪改版的正確啟動方式

昨晚開啟的 WMOM-20260513-01 是一個刻意保留空白的 placeholder。這個設計決策同樣值得記錄：沒有 spec 就不開工。

在許多開發文化中，「先動起來再說」是一種美德。但在工業 B2B 軟體的 UI 設計中，過早開始意味著開發出難以回滾的 UI 債務。windMindOM 的 issue 系統對此有明確的規範：每一個 UI 改版 issue 必須包含設計交接書、異動範圍說明、Acceptance Criteria，缺一不可。

🔍 windMindOM UI v2 目前已知的方向

基於昨晚的 commit 記錄，v2 改版的觸發點是劉老師在 2026-05-12 提出了「新一輪 UI 改版」的意向。目前 WMOM-20260513-01 的 Acceptance Criteria 框架已備好，等待補充：新的主題風格方向、字型與色系調整範圍、元件異動清單（哪些沿用 v1，哪些需要重新設計）。v1 的六套設計變體史料將是 v2 設計者最重要的參考資產。

四、研究室的設計工程實踐啟示

windMindOM 的設計管理實踐，對研究室未來的所有軟體專案都有廣泛的參考價值。這裡整理幾個核心原則：

原則一：設計決策是工程知識，必須版本控制。 JSX 元件、CSS 變數、設計交接書，都應和功能程式碼一起存在 Git 中。設計不是暫時的輸入，而是永久的工程資產。

原則二：多變體探索優於單一方案。 VA/VB/VC 的探索方式，讓設計決策有了比較基礎。「為什麼選 VB 而非 VA？」有了可引用的具體答案。在時程允許的情況下，寧可多花半天產出三個變體，也不要讓單一方案在沒有對照下被接受。

原則三：placeholder issue 優於提前開工。 當需求模糊時，開一個有框架的 placeholder，比在不完整 spec 下開始動手要好得多。等 spec 補齊，再啟動實作 sprint，效率反而更高。

原則四：以 LLM 為設計協作者時，context 即史料。 設計檔案的完整保存，讓 LLM 在下一輪改版時能夠讀取完整的設計脈絡，而非從頭描述。這是 AI 協作設計的新基礎設施思維。

🚀 研究室接下來的行動項目

windMindOM：等待劉老師補充 WMOM-20260513-01 的設計交接書，正式啟動 UI v2 改版 sprint。在此之前，M4 剩餘工作（A6/A7 領料庫存 frontend、A10 E2E test）繼續推進。

設計文化推廣：考慮將「設計史料歸檔」實踐推廣至 windAILab 與 RAG_Ultimate 的前端元件，建立統一的 docs/design/ 規範。

LLM 設計協作 SOP：根據 windMindOM 的實踐，整理一份「如何讓 Claude 讀取設計史料進行 UI 改版」的標準作業流程，供研究室未來專案參考。

工業軟體的介面設計，從來不是「美不美觀」的問題，而是「安不安全、用不用得動、改得了嗎」的工程問題。windMindOM 選擇把設計史料當作工程資產，正是這種理念的具體實踐。
