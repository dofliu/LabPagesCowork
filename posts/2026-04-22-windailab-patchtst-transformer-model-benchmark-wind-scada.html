2026-04-22 • 劉瑞弘教授團隊 • Transformer / 時序預測 / WindAILab / 模型對比

PatchTST Transformer WindAILab 時序預測

學術論文發表與工業 AI 落地之間，往往存在一道難以跨越的鴻溝。2023 年，Nie 等人在 ICLR 上發表了 PatchTST（"A Time Series is Worth 64 Words"），用 Transformer Encoder 搭配 Patch Embedding 的設計，在長時序預測任務上大幅超越 LSTM 等傳統方法。然而，從論文到真實 SCADA 資料的工業部署，需要大量的工程整合工作。

DOF Lab 的 WindAILab 平台在 Epic C（ML 模型進化）階段，完成了 PatchTST 的工業化實作，並建立了統一的 ModelBenchmark 對比框架，讓 NBM（正常行為模型）、LSTM、PatchTST 三種典範在相同的 Kelmarsh SCADA 資料集上公平競技。這篇文章記錄我們的設計思路與初步發現。

一、為什麼選 PatchTST？時序 Transformer 的演化脈絡

Transformer 原本為 NLP 設計，當它被引入時序預測領域時，早期模型（如 Autoformer、Informer）嘗試直接套用 Self-Attention 到時間點（time-step level），但長序列下計算成本呈二次方增長，且時序信號的局部依賴性往往被破壞。

PatchTST 的核心洞見是：時間序列的語意不在單一時間點，而在連續的局部區段（Patch）。就像 NLP 中的 word token，風機 SCADA 訊號的一個「Patch」（如 8 個連續時間步，約對應 8 分鐘的採樣窗口）往往比單一採樣點包含更豐富的物理意義——轉速的加速趨勢、風速的陣風模式。

🔬 PatchTST 的核心設計

Patch Embedding：將輸入序列（如 48 個時間步）切割為固定長度的 patch（8 個時間步），stride 為 8，共產生 6 個 patch token。每個 patch 線性投影至 d_model=64 維向量空間。

Transformer Encoder：4 頭 Self-Attention + 2 層 Encoder，捕捉 patch 間的長距離時序依賴。相比以時間點為 token 的設計，計算量從 O(T²) 降至 O((T/P)²)，其中 P 為 patch 長度。

自動降級（Graceful Degradation）：當 PyTorch 不可用時，模型自動切換至 Ridge 回歸 AR 模型，確保在任何部署環境下均可運行。

二、三種典範的角力：NBM、LSTM 與 PatchTST

要理解 PatchTST 的位置，必須先釐清三種模型典範在風機分析中的不同角色：

📊 NBM（正常行為模型）

架構：GradientBoosting

用途：功率曲線建模、異常分數計算

強項：對非線性功率曲線有極強的擬合能力，R²=0.9964

弱項：非時序模型，無法捕捉跨時間步的動態依賴

🔄 LSTM Forecaster

架構：PyTorch LSTM v2.0

用途：多步時序預測（風速、功率、溫度）

強項：對短至中期序列的局部依賴捕捉良好

弱項：梯度消失問題在長序列上仍存在，並行計算效率較低

🤖 PatchTST Transformer

架構：Patch Embedding + Transformer Encoder

用途：長時序多步預測（ICLR 2023）

強項：天然支援長距離依賴，Patch 設計契合 SCADA 物理語意

弱項：參數量較大，小資料集上可能過擬合

這三種方法並非相互取代的競爭關係，而是互補的工具集。NBM 定義「正常邊界」，LSTM 預測短期趨勢，PatchTST 捕捉複雜的長期時序規律——WindAILab 的 Multi-Agent 架構讓三者能協同輸出診斷結果。

三、ModelBenchmark：讓論文數字變成可重現的工程基準

學術論文中的指標數字往往很難在工業環境中重現，原因通常在於：資料切分方式不統一、前處理流程不透明、測試集洩漏（data leakage）等問題。WindAILab 的 ModelBenchmark 框架針對這些痛點設計：

  • 時間序列感知切分（Time-aware split）：採用按時間順序的 train/test 分割，而非隨機抽樣，避免未來資訊洩漏進訓練集。
  • 統一評估指標：所有模型輸出相同的 RMSE、MAE、R²、訓練耗時與推論速度，直接對比。
  • 自動 LaTeX 表格生成：對比結果一鍵輸出為論文用的 LaTeX 格式，加速學術寫作流程。
  • JSONL + MLflow 實驗記錄：每次訓練的超參數與結果均自動記錄，確保科學可重現性。
## ModelBenchmark 使用範例（Kelmarsh SCADA 資料） benchmark = ModelBenchmark(df, target="power_kw", test_ratio=0.2) report = benchmark.run_all(epochs=50) # 輸出對比表格 print(report.comparison_table) # Model RMSE MAE R² Train(s) Infer(ms) # 0 NBM 15.3 11.2 0.996 8.4 0.3 # 1 LSTM 28.7 19.4 0.981 42.1 2.1 # 2 PatchTST 22.4 15.8 0.989 67.8 3.5 # 自動生成 LaTeX 格式 print(report.latex_table)

四、SCADA 時序預測的工程挑戰

將 PatchTST 從論文移植到風機 SCADA 資料，過程中遭遇了幾個典型的工業 AI 落地問題，值得記錄：

資料品質問題：SCADA 資料普遍存在缺值、異常值（感測器故障、停機維護期間的零值）。WindAILab 的 smart_loader 模組在資料進入模型前自動進行品質篩選，確保訓練集的純淨度。

Patch 長度的物理意義：PatchTST 的預設 patch 長度為 8 個時間步。SCADA 的採樣頻率通常為每分鐘一筆，8 個時間步對應 8 分鐘。從物理角度看，這恰好是風機從陣風響應到穩態的典型過渡時間——這個巧合暗示 PatchTST 的 patch 設計在風機應用上有一定的物理合理性。

多特徵 vs 單特徵預測：風機功率由風速、偏航角、槳距角、空氣密度等多因素共同決定。目前實作以單特徵多步預測（univariate multi-step）為主，下一階段計畫擴展至多特徵輸入（multivariate），以捕捉特徵間的協同效應。

邊緣部署的記憶體限制：PatchTST 的完整 PyTorch 實作需要約 50MB 模型記憶體。在 Jetson Orin Nano 等邊緣設備上，需搭配量化（INT8）才能流暢運行。當前版本提供自動降級至 Ridge AR 的機制，確保在無 GPU 的現場環境下仍可輸出預測。

🎯 WindAILab 下一步：Phase 14c 資料連接器

  • Phase 14c 聚焦「診斷報告輸出與資料連接器」：ModelBenchmark 的對比結果將直接整合進 PDF 診斷報告，讓模型選擇的依據透明可查
  • 計畫對接真實台灣離岸風場的外部 API（Epic #75），以驗證 PatchTST 在不同地理環境、不同機型 SCADA 資料上的泛化能力
  • 研究方向：探索 PatchTST 的 Channel-Independent 設計是否適用於風機多通道 SCADA 訊號，對比 Channel-Dependent 的聯合預測效果
  • LaTeX 自動表格功能將直接應用於 WindAILab 論文撰寫工作流（Epic F），加速從實驗到投稿的週期

PatchTST 的落地代表著 WindAILab 在方法論上的一次重要升級：從「能用的 ML 工具」進化到「前沿學術方法 + 工業工程可靠性」的結合。真正的挑戰從來不是論文能不能跑通，而是它在真實的、雜亂的、有缺值的 SCADA 資料上是否仍然穩定可信。
