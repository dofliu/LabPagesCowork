研究平台也要生產品質：RAG_Ultimate 基礎建設強化的 90 輪例行實踐

2026-05-17 • 劉瑞弘教授團隊 • RAG / DevOps / Code Quality / Research Engineering

RAG_Ultimate DevOps Code Quality Research Engineering CI/CD

研究室的程式碼通常有個隱性預設：「只要實驗能跑、論文能寫，工程品質是次要的。」這個預設在研究早期或許合理，但當平台規模擴大到 17 種 RAG 檢索模式、829 支自動化測試、多個前後端服務同時運作時，品質欠債就會以指數速度累積。

RAG_Ultimate 在過去幾個月採取了一個不同的策略：透過「例行品質電池（Routine Quality Battery）」機制，以每次一小塊、不超過千行程式碼的節奏，持續對平台進行工程強化。截至今日（2026-05-17），我們已累積超過 90 輪排程觸發、合併超過 100 個 Pull Request。這篇文章回顧這週剛完成的基礎建設強化週期（PR #96–#101），以及背後的工程哲學。

這週完成了什麼：基礎建設強化五連發

本週的例行輪次聚焦在「讓平台更可觀測、更可部署」，完成了以下五個環環相扣的改善：

PR #96
後端集中式 Logging 配置
新增 app/core/logging.py 模組，加上 RequestContextMiddleware，所有請求進出皆自動附帶 request_id 與時間戳記，讓 debug 時不再在 print 語句海中大海撈針。
可觀測性
PR #97
Health Check 拆分：Liveness vs. Readiness
/health 端點從 main.py 抽離為獨立模組，並明確區分「活體檢查（liveness）」與「就緒檢查（readiness）」。這是 Kubernetes / Cloud Run 部署的標準要求——前者確認服務未當機，後者確認資料庫連線等依賴項均已就緒。
可部署性
PR #98
Docker Compose：Qdrant Healthcheck + 具名 Volume
在 docker-compose.yml 加入 Qdrant 服務的 healthcheck 設定，FastAPI 容器改為 depends_on: {condition: service_healthy}，確保向量資料庫完全就緒後才啟動 API 服務；同時將匿名 volume 改為具名 volume，解決每次 docker compose down 後資料消失的問題。
容器化資料持久性
PR #99
API Base URL 環境變數強型別化
前端所有服務的 API 端點 URL 統一改用 Vite 的 import.meta.env，並補上 ImportMetaEnv 型別宣告，消除散落在各元件的硬編碼 localhost:8000。此後切換開發、測試、生產環境只需修改 .env 一個檔案。
多環境支援
PR #100
Backlog 補完 + ts-node 開發依賴
新增 ts-node 作為 devDependency，解鎖後續可直接執行 TypeScript 腳本的能力；同時整理並新增四個 backlog 項目（H1–H3 程式碼健康度修補 + G7），確保品質債的能見度。
開發工具鏈

🔬 為什麼這些改善對研究平台同樣重要？

Liveness/Readiness 區分、具名 Docker Volume、環境變數管理——這些聽起來像是 SRE 或 DevOps 工程師才需要煩惱的事。但當研究助理試圖在新電腦重建實驗環境，或者我們需要在 GCP Cloud Run 上跑完整的 RAG benchmark 時，缺少這些基礎設施就意味著數小時的環境除錯，而非數分鐘的部署。可重現的研究環境，本質上就是生產工程的一個子集。

例行品質電池：小步快跑的工程哲學

RAG_Ultimate 採用的「例行品質電池」並不是一次性的技術債清償計畫，而是一個持續運作的自動化排程機制。每次觸發遵循嚴格的規則：

📏 規模上限
每輪 PR 不超過 1000 行、不超過 10 個檔案。超過上限就拆分，本輪只做第一塊。這確保每個 PR 都能快速 review，也讓 git blame 在未來依然有意義。
🚦 禁區制度
Backlog 明確標記「禁區外（🟢）」「邊界（🟡）」「純禁區（🔴）」。自動輪次只能動禁區外的項目，涉及核心服務邏輯的改動必須等使用者授權，避免自動化過程引入不預期的功能變化。
🔍 品質電池
每五個 PR 後自動執行一輪 CR/CI checkpoint：tsc 型別檢查、jest 單元測試、ruff 靜態分析、pytest 後端測試全套跑完，結果記錄在 routine_log.md，形成可追溯的品質曲線。
⏸ 等待機制
若有自己建立的 open PR 尚未 merge，下一輪自動跳過（skip），並記錄連續等待輪次。歷史紀錄顯示最長曾連等 5 輪——這反映了人工 review 的節奏，也是系統刻意設計的「人機協作卡點」。

這個機制的設計靈感來自持續交付（Continuous Delivery）的核心思想：與其積累大型變更後一次性合併，不如每次只做一件確定的事，讓主線隨時保持可部署狀態。對於研究平台而言，「可部署」等同於「實驗可重現」。

當自動化遇上 PR Review 瓶頸

這週有個有趣的現象：PR #101（CR/CI checkpoint #4 文件）在建立後連續 5 輪排程觸發都處於「等待 merge」的狀態。從 routine_log 可以清楚看到系統的行為：

- run_id: 2026-05-17-r5 status: skipped task_ref: "等候 PR #101 merge" notes: | 連續 skip 統計：r4 + r5(=本輪) = 2 連 skip 歷史對照： PR #99 等候期：5 連 skip（歷史最高） PR #100 等候期：4 連 skip 建議使用者執行： gh pr merge 101 --squash --delete-branch --repo dofliu/RAG_Ultimate

系統不會自行 merge PR，也不會在等待過程中做任何未授權的改動——它只是忠實地記錄狀況，並在每輪 log 中明確提示人類應採取的行動。這種「明確等待、不擅自行動」的設計，在 AI 輔助開發工作流中至關重要：自動化應該加速工作，而不是繞過人類的判斷。

💡 研究啟示：AI 輔助程式碼維護的邊界設計

RAG_Ultimate 的例行品質電池提供了一個具體的案例：AI 代理可以安全地在明確邊界內自主執行任務，但關鍵決策（merge、授權進入禁區、調整研究方向）仍應由人類把關。這種「可信邊界（Trusted Boundary）」設計將是下一代 AI 輔助研究系統的核心議題，也是我們在研究室日常工作中正在積累的實踐知識。

下一步：從基礎建設到研究主線

PR #101 merge 後，例行品質電池的下一個目標是：

  • H2：修復 jest haste collision（jest.config.ts 過濾 worktree 路徑，清除本機測試紅字警告）
  • C1：backendApiService.ts 與後端 schemas.py 完整型別對齊
  • C3/C4：前端 fetch 重試 middleware + 全域 toast 提示

完成這批基礎建設後，RAG_Ultimate 將正式進入下一個研究主線：Phase 2 後端 ingest/embed/retrieve API 的完整實作，以及 Phase 3 的風場文件 RAG benchmark 論文研究。

有趣的是，正是因為前 90 輪例行維護把 TypeScript 嚴格化（tsc 0 errors）、測試覆蓋建立（829 tests）、CI/CD 部署自動化，我們才有信心在接下來的研究主線中快速迭代——知道每一步都有安全網撐著。
