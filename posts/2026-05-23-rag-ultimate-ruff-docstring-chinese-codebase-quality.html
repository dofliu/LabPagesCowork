自動化品質管理的邊界：當 ruff 遇上中文 docstring 的不可解張力

2026-05-23 • 劉瑞弘教授團隊 • Python / Code Quality / Research Engineering / i18n

RAG_Ultimate Routine Cycle #6 Code Quality ruff / linting i18n Research Code Docstring Engineering

在 AI 研究程式碼庫的日常維護中，有一個問題很少被公開討論：當你的程式碼用中文撰寫 docstring，但程式碼品質工具是為英文設計的，自動化修復應該走到哪裡，又應該在哪裡停下？

2026 年 5 月 22 日，RAG_Ultimate 的 Routine 系統完成了 Cycle #6 的第三個 PR（#132）：使用 ruff 自動修補 12 處 D 系列 docstring 警告。這看起來是一個平凡的程式碼品質任務，但在執行過程中，routine 做出了幾個值得深思的技術決策——哪些警告可以安全自動修、哪些必須保留給人類判斷、哪些根本無法在不破壞中文一致性的情況下自動處理。這些決策，折射出研究型程式碼庫在「可自動化品質管理」邊界上的獨特挑戰。

一、PR #132 的技術決策地圖

執行 ruff --select D --fix 聽起來很簡單，但 RAG_Ultimate 的 routine 在執行前，仔細分類了所有 D 系列（pydocstyle）規則，得出一份「可以修 / 不能修 / 需討論」的決策地圖：

規則問題描述本輪決策理由
D212 多行 docstring 的摘要應從第一行開始（不是第二行） 自動修 ×7 純格式問題，不涉及語意，ruff --fix 安全可靠
D403 單行 docstring 首字應大寫 自動修 ×2 英文首字大寫是通用慣例，自動修不影響語意與中文段落
其他 D 系列 各類格式問題 自動修 ×3 安全類格式問題，批次修補
D407 / D413 section 標題後應有破折線分隔（numpy-style） 明確 --ignore 已存在 Google-style docstring（Args: 後無破折線）；強制修改會造成風格不一致
D400 / D415 docstring 應以英文句點 . 結尾 永久跳過（需人類決策） 140 處中文 docstring 以全形句號 。 結尾——自動修會把 。 改成 .，破壞中文語言一致性

這 5 類規則的決策，構成了一個「可自動化程度」的光譜：從完全安全的格式調整，到需要技術判斷的風格選擇，再到涉及語言本質的不可自動化問題。

📋 PR #132 實際執行指令

# 對 scripts/ 與 RAG_Enterprise_Backend/app/api/endpoints/ 執行 ruff --select D --ignore D407,D413 --fix . # 驗證結果 ruff check → All checks passed! # D 系列警告從 169 降至 157 pytest test_benchmark_ingest.py → 41 passed in 0.12s tsc --noEmit → 0 errors # 前端型別安全無變化 jest → 89 suites pass # +2 pre-existing legacy (與本 PR 無關)

二、D400/D415 與中文句號：一個無法用工具解決的問題

最有趣的案例是 D400 和 D415。這兩條規則要求 docstring 以英文句點 . 結尾——這在全英文程式碼庫中是完全合理的慣例，但在 RAG_Ultimate 這樣的中英混用研究程式碼庫中，卻形成了一個根本性的矛盾。

RAG_Ultimate 的 API endpoints 和 scripts 中，有 140 個 docstring 以中文全形句號 。 結尾，例如：

def get_collection_stats(collection_name: str) -> dict: """取得指定集合的向量統計資訊。""" # D400/D415: 應以 . 結尾 — 但這個 。 是正確的中文標點

如果強制執行 ruff --unsafe-fixes，它會把 。 改成 .，產出：

"""取得指定集合的向量統計資訊.""" # ↑ 中文句子以 ASCII 英文句點結尾 — 語法上「合規」，閱讀上很奇怪

這 140 個警告，因此進入了「routine 永遠跳過」清單。它們在技術上是 ruff 的警告，但在語言使用上沒有任何問題。正確的解法不是自動修補，而是在 pyproject.toml 的 ruff 設定中加入規則例外，或改用 convention = "google" convention 整體切換。但這個決策涉及程式碼庫的文件風格政策，不應由自動化代理單方面決定——routine 因此把這個問題轉成了一個 GitHub issue，等待研究者與學生團隊討論。

💡 洞見：工具的「合規」與語言的「正確」是兩回事

程式碼品質工具的規則，幾乎全部假設程式碼的自然語言部分是英文。在全球化的研究環境中，這個假設越來越不成立。中文、日文、韓文的研究程式碼庫日益增多，但 ruff、pylint、flake8 的 docstring 規則並未針對非 ASCII 語言設計。這不是工具的 bug，而是工具的設計邊界——識別這個邊界，比強行繞過它更重要。RAG_Ultimate routine 的做法是：把 140 個「工具認為不合規但語言上完全正確」的警告，明確標記為「需人類決策的設計問題」，而非技術債。

三、風格一致性：Google-style vs. numpy-style 的選擇困境

另一個有趣的決策是 D407/D413 的 --ignore。這兩條規則要求 section 標題（如 Args:、Returns:）後應有破折線分隔——這是 numpy-style docstring 的格式要求。但 RAG_Ultimate 在 services/ 等核心模組中，已有大量 Google-style docstring：

Google-style（現有）
Args: query: 查詢字串 top_k: 返回數量 Returns: 檢索結果列表
numpy-style（D407/D413 要求）
Args ---- query : str 查詢字串 top_k : int 返回數量

若在不明確設定 convention = "google" 的情況下啟用 D407/D413 自動修，ruff 會把所有 Google-style 的 Args: 後面加上破折線，造成與 services/ 現有格式的風格衝突。更糟糕的是，這個改動在功能上完全無害，但在 code review 時會引發大量雜訊，分散對實質內容的注意力。

Routine 的決策很簡單：沒有明確選型就不動。在 pyproject.toml 明確設定 docstring convention 之前，D407/D413 維持在 ignore 清單，避免製造人為的風格分裂。

四、給研究型程式碼庫的品質管理建議

RAG_Ultimate PR #132 的經驗，提供了幾個對中文研究型程式碼庫有參考價值的品質管理原則：

  • 分層管理 linting 規則：把規則分成「安全自動修」、「需要技術判斷的忽略」、「需要人類決策的設計問題」三層，而不是全部啟用或全部關閉。
  • 在 pyproject.toml 明確宣告 docstring convention：選定 Google-style 或 numpy-style，在設定檔中明確寫出，讓工具和人類都清楚基準是什麼。
  • 語言本質問題轉成設計討論：對於 D400/D415 這類與語言本質衝突的規則，不要用 noqa 一行一行壓制，應在 pyproject.toml 的 [tool.ruff.lint] 中統一 ignore，並在 issue tracker 留下為什麼這樣做的理由。
  • 自動修只做無損操作：每次自動修補後都應完整跑測試套件（ruff check + pytest + tsc），確認沒有任何行為變化，只有格式改善。
  • 記錄「刻意不修」的原因：在 commit message 或 PR 描述中明確說明哪些警告被忽略、為什麼，讓未來的維護者（和 AI 代理）理解決策脈絡。

⚠️ 警示：不加思索的「全部修」可能比「不修」更糟

自動化品質工具最危險的使用方式，是在沒有理解規則意涵的情況下執行 --fix --unsafe-fixes 全量修補。在中文研究程式碼庫中，這可能把 140 個語言正確的中文句號改成語感奇怪的英文句點、把精心選定的 Google-style docstring 改成 numpy-style、或在 services/ 等禁止自動修改的核心模組中引入不必要的 diff。自動化的價值在於一致執行人類已設計好的規則，而非替代人類進行風格設計決策。

程式碼品質管理的藝術，不在於讓所有警告歸零，而在於清楚知道：哪些警告代表真正的問題、哪些代表工具的設計假設與你的使用情境不符。RAG_Ultimate Routine 在 PR #132 中的決策，為中文研究型程式碼庫提供了一個具體的參考案例：自動化應該做它最擅長的事（一致執行安全規則），而把真正需要判斷的問題交還給人類（語言政策、風格選型、套件授權）。
