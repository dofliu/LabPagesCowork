2026-05-21 • 劉瑞弘教授團隊 • RAG / Wind Energy / Benchmark / Research Infrastructure

RAG_Ultimate Phase 3 Wind Farm RAG Benchmark Research Pipeline Paper

建造一棟研究大樓，最難的一關往往不是地基或鋼架，而是那個微妙的移交時刻：工程部門將鑰匙遞給研究部門，宣告「結構已竣工，現在輪到你們了。」

2026 年 5 月 21 日，RAG Ultimate 平台正式站在這個移交時刻之上。歷經四個多月、超過 120 個 Pull Request、22 項 routine backlog 全部勾完，平台從一個拼拼湊湊的研究腳本集合，演化成一個具備完整 FastAPI 後端、17 種 RAG 檢索模式、Prometheus 指標、Cloud Run 部署管線與 Sentry/OpenTelemetry 可觀測性 stub 的企業級研究環境。工程里程碑 Phase 2 全面完工。下一站：Phase 3 — 風電 RAG Benchmark 論文。

一、從 10 個散落腳本到一個統一平台：v1.0 → v8.0 的整合歷程

RAG Ultimate 的前身是研究室分散在各個資料夾裡的原型實驗：一個切片實驗室、一個評測腳本、一個微調介面、幾個 RAG 策略的 Jupyter Notebook。它們各自有效，卻無法協同。每次要做比較實驗，研究生都需要在不同環境間搬移資料，手動整理結果。

二、Phase 2 遺留的基礎建設：為什麼論文研究需要工程底座？

外界看 RAG 研究，往往聚焦在演算法和數據集。但要讓研究可重複、可擴展、可發表，底層基礎建設至關重要。Phase 2 在過去幾週密集交付的七大元件，正是為 Phase 3 論文研究奠定可信的實驗環境：

🔌 七大後端 API Endpoint
/ingest、/embed、/retrieve、/collections、/backup、/metrics、/health——涵蓋文件攝取到向量存取的完整生命週期，讓每次 RAG 實驗都通過統一介面執行，結果具有可比性。
📊 Prometheus 指標 + 觀測 Stub
PR #109 交付 Prometheus 格式指標端點；PR #115 植入 Sentry + OpenTelemetry stub。實驗運行時的 QPS、延遲分布與錯誤率，將有完整的追蹤記錄——這對論文的「實驗條件重現」至關重要。
☁️ Cloud Run 部署管線
多環境 Dockerfile + Cloud Build 設定（PR #114）讓 benchmark 可以選擇從本地開發環境或 GCP staging 環境執行。對千份風電文件的大規模 benchmark，雲端執行路徑可顯著縮短實驗週期。
🧪 829 個自動化測試
41 個測試 suite、829 個案例，涵蓋 tsc 0 錯誤、ruff 全綠。每次修改 RAG 策略邏輯時，回歸測試套件都能保護既有的實驗結果不因工程變更而悄悄失效。

💡 研究工程學的核心洞見

在工業界，工程基礎建設（CI/CD、監控、測試）是為了讓產品穩定運行。在學術研究中，同樣的基礎建設扮演著不同但同等重要的角色：確保實驗結果的可信度與可重現性。一個沒有回歸測試保護的 RAG 研究系統，每次修改策略都可能無意間改變其他模式的行為，使跨模式比較結果失去意義。Phase 2 的工程投入，本質上是在為 Phase 3 論文的「實驗設計完整性」投資。

三、Phase 3 的實際任務：風電 RAG Benchmark 的設計與挑戰

Phase 3 的核心目標是回答一個對風能產業有實際意義的研究問題：在風電技術文件的特定語境下，哪種 RAG 檢索策略能提供最準確、最具可解釋性的答案？

目前的 Benchmark 設計分為兩個 Tier：

Tier說明目前狀態
Tier-1 12 種核心 RAG 策略主比較（Naive / Sentence Window / HyDE / RAPTOR / Cross-Encoder / Hybrid BM25 等），每種策略跑 5 runs 取平均，以 RAGAS 四項指標評測 修正中：初跑完成，Rerank / Cross-Encoder / Hybrid / HyDE 有 104 個 errors，待修正後重跑
Tier-2 5-run aggregate 匯總；部分 mode 因本地網路斷線被標為「不可信」，待補跑 r06、r07 補跑中：9/12 mode 需補跑確認

風電 RAG 的特殊挑戰在於文件語境的高度專業性。風機操作手冊、故障診斷指引、IEC 61400 標準文件，充滿大量技術縮寫、物理量符號與跨文件的隱性依存關係。通用 RAG 策略在這種語境下往往表現不穩定——這正是這個 Benchmark 研究的學術貢獻所在。

🌬️ 風電 RAG Benchmark 的研究設計重點

  • 文件語料庫：風機操作手冊、SCADA 異常報告、IEC 61400 標準節錄
  • 評測指標（RAGAS）：Faithfulness（忠實度）、Answer Relevancy（答案相關性）、Context Precision（上下文精度）、Context Recall（上下文召回率）
  • QA 評測集：405 道問題，涵蓋故障診斷、維護程序、技術規格查詢三大類型
  • 嵌入模型對比：Gemini text-embedding-004（1536 維）vs. sentence-transformers 本地模型
  • 向量庫：Qdrant persistent（支援離線重用實驗向量，避免重複計算）

四、ROADMAP 對齊：PR #122 為何重要

今日合入的 PR #121（README 整合狀態表）與待 merge 的 PR #122（ROADMAP 對齊），看似只是文件更新，實則代表一個重要的認知同步動作。

研究系統在快速迭代過程中，文件往往落後於實作。開發者的腦中雖然清楚「Phase 2 已完成、Phase 3 是下一步」，但若 ROADMAP.md 仍然寫著過時的狀態，對新加入的研究生或合作者而言，系統的現狀與方向就會模糊不清。PR #122 透過更新 ROADMAP 中 backlog 21/22 的完成狀態，以及清理已完成 routine 的佔位日誌，讓整個平台的文件狀態與程式碼狀態達成一致。

⚠️ 當 Routine 遇到人工決策瓶頸：7 連 Skip 的啟示

PR #122 目前已通過 CI（零測試失敗），但等待人工 merge 超過一天，導致自動化 routine 連續 7 輪記錄為「skipped」。這個現象本身就是一個值得反思的工程管理課題：自動化工具能接力工程執行，但無法替代人工的決策點。PR review 與 merge 是研究者對程式碼變更的認可，是知識工作中不可被自動化省略的環節。Routine 的「skip streak」不是系統失敗，而是對人工介入的如實等待與記錄。
