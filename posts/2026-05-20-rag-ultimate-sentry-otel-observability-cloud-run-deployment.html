從研究工具到可觀測生產服務：Sentry + OpenTelemetry 為 RAG 平台注入錯誤追蹤與分散式追蹤能力

2026-05-20 • 劉瑞弘教授團隊 • RAG / Observability / Sentry / OpenTelemetry / Cloud Run

RAG_Ultimate OpenTelemetry Sentry Cloud Run Observability Production Engineering

一個研究系統能「跑起來」與能「被信任地運行」之間，存在著一段關鍵的距離。這段距離的名字，叫做可觀測性（Observability）。

過去幾週，RAG_Ultimate 平台密集地完成了型別系統對齊（PR #110）、Prometheus 指標端點（PR #109）、全域 async toast 機制（PR #112）、千份文件吞吐量基準測試骨架（PR #113），以及多環境 Dockerfile 與 Cloud Build 部署管線（PR #114）。這一連串工程動作，有一個共同的指向：讓這個起初作為風電 RAG 論文研究工具的系統，具備真正意義上的生產等級可觀測性。今日，PR #115 正式將 Sentry 錯誤追蹤與 OpenTelemetry 分散式追蹤的骨架（stub）植入後端，是這趟可觀測性工程旅途中的關鍵里程碑。

一、可觀測性三支柱與研究系統的特殊挑戰

現代生產系統的可觀測性建立在三個支柱之上：Metrics（指標）、Logs（日誌）、Traces（追蹤）。三者各有所長，分別回答不同層次的問題。

📊 可觀測性三支柱：各自回答什麼問題？

支柱回答的問題RAG_Ultimate 的實作
Metrics 系統整體健康狀態如何？QPS 是多少？延遲分布如何？ 已完成 PR #109 — GET /api/v1/metrics（Prometheus 格式）
Logs 這個請求發生了什麼事？哪一行程式拋出了例外？ 已完成 結構化 logging（FastAPI middleware）+ Sentry F1a stub
Traces 這個請求跨越了哪些服務？哪個環節是效能瓶頸？ 待 merge PR #115 — OpenTelemetry stub（F1a），F1b 完整實作待授權

研究型系統在可觀測性方面面臨一個特殊挑戰：它往往在功能層快速迭代，但基礎設施層的投資卻容易被推遲——「等論文截稿後再說」。然而，當系統開始承載真實的風電文件 RAG benchmark 資料（405 題 QA 集、Qdrant persistent 向量庫），或即將整合 windMindOM 的知識模組（M5 計畫），任何未被觀測到的例外與效能退化都可能悄悄扭曲研究結論。

二、Sentry：讓每一個例外都留下案底

Sentry 是業界最廣泛使用的即時錯誤追蹤平台。它的核心機制很直觀：在應用程式的 exception handler 層注入一個 SDK，每當未被捕捉的例外（或明確呼叫 sentry_sdk.capture_exception()）發生時，Sentry 自動收集：

  • 完整的 stack trace（含每一層的原始碼片段）
  • 觸發例外的 HTTP 請求資訊（URL、headers、request body）
  • 環境標籤（staging vs. production、版本號、Git commit SHA）
  • 用戶 context（在 RAG 平台中可記錄是哪個 benchmark 集觸發的查詢）
  • breadcrumbs（例外發生前的事件序列，類似飛機的黑盒子紀錄）

💡 為什麼研究系統特別需要 Sentry？

研究型 RAG 系統的邊界條件往往藏在特定的輸入組合中——例如某份特殊格式的風電維修報告觸發了 embedding 模型的長度截斷，或某個 Qdrant filter 的邊界值產生了意料外的空結果。這些問題在正常的 benchmark 測試中可能不會出現，卻在真實使用者查詢時突然浮現。Sentry 讓每一個這樣的事件自動留下可追查的紀錄，而不是消失在 Docker container log 的無盡滾動中。

PR #115（F1a stub）採用的是「骨架先行」策略：在 main.py 的 FastAPI lifespan 中加入 sentry_sdk.init() 呼叫的占位符，以環境變數 SENTRY_DSN 控制是否啟用，並在 requirements.txt 中標注 sentry-sdk[fastapi] 為待安裝依賴。這樣的設計讓程式碼在未安裝 SDK 時能正常跑通 CI，同時為 F1b 完整實作（預估約 70 行）預留清晰的插入點。

三、OpenTelemetry：分散式追蹤的開放標準

如果說 Sentry 是「事後查案」的工具，OpenTelemetry（簡稱 OTel）則是「即時跟蹤」的架構。OTel 是 CNCF（雲端原生運算基金會）主導的可觀測性開放標準，整合了過去分散的 OpenTracing 與 OpenCensus 兩大規範，目標是讓一套 instrumentation 程式碼能輸出到任意後端（Jaeger、Zipkin、Grafana Tempo、Google Cloud Trace……）。

🔬 Span：追蹤的基本單位
每個 Span 代表一個操作的時間段落，帶有名稱、開始/結束時間戳、以及任意的 key-value attributes。多個 Span 組成一條 Trace，描述一個請求從進入系統到返回結果的完整路徑。
🔗 Context Propagation：跨服務追蹤
當 RAG_Ultimate 的 FastAPI 後端呼叫 Qdrant 向量檢索時，OTel 的 context propagation 機制確保同一條 trace ID 能貫穿整個呼叫鏈，讓「embedding → retrieval → reranking → LLM」的延遲分布一目了然。

對 RAG 研究系統而言，OTel 的價值不只是找效能瓶頸。當我們同時評估 17 種檢索模式（dense、sparse、hybrid、HyDE、RAPTOR……）時，能夠精確比較每種模式在 retrieval 階段的延遲分布，是論文中 latency-recall trade-off 分析的關鍵資料來源。這比單純記錄 end-to-end 延遲更具研究意義。

四、多環境部署管線：PR #114 的架構決策

可觀測性工具的價值，必須建立在有可靠部署管線的前提下——你無法在一個每次部署都要手動操作的系統上，有效運行自動化的錯誤追蹤。PR #114（昨日 merged）正是解決這個前提問題。

🚀 多環境 Dockerfile + Cloud Build 部署架構

feature branch
→
PR merge to main
→
Cloud Build trigger
→
Build multi-stage Docker image
→
Push to Artifact Registry
→
Deploy to Cloud Run (staging)
→
Manual approval → production

多階段 Dockerfile 分離 builder（含開發依賴）與 runtime（精簡映像），staging 環境自動部署、production 需手動核准，以 Cloud Build substitution variables 管理環境差異。

這個部署架構有三個值得關注的設計選擇。第一，使用多階段 Dockerfile（multi-stage build）將安裝測試套件的 builder stage 與實際部署的 runtime stage 分離，讓 production 映像大幅縮小，同時確保 CI 測試環境與本機開發環境的一致性。第二，採用 Cloud Build substitution variables（而非多個 Dockerfile）管理 staging/production 差異，遵循「單一可信來源」原則，避免環境設定漂移。第三，staging 自動部署、production 手動核准的雙軌機制，在自動化速度與人工把關之間取得了適合研究系統的平衡點。

五、自動化 Routine 系統的品質保證哲學

RAG_Ultimate 近期的高頻更新（今日已累積至 r55 輪次），背後依賴一套名為 Routine 的自動化品質維護機制。這個機制的核心規則很簡單：每次只做一件事，做完就開 PR，有未 merge 的自建 PR 就停下來等待。

PR #115 目前 CI 全綠（ruff + pytest + tsc 三重通過）、mergeable 狀態確認，正在等待人工 merge。這個「等待」本身也是 Routine 設計哲學的體現：系統確保不會在舊工作未確認前就疊加新的修改，防止技術債在自動化節奏中悄悄累積。

結語：可觀測性是研究誠信的基礎建設

在研究領域，「可重現性（Reproducibility）」是最基本的學術誠信要求。但在系統研究中，可重現性不只是「相同輸入得到相同輸出」，還包括能夠在系統出現非預期行為時，準確追溯發生了什麼。

Sentry 讓每個例外都留下可查詢的紀錄；OpenTelemetry 讓每個請求的路徑都變得透明；Prometheus metrics 讓系統整體健康狀態隨時可量化；多環境部署管線確保每次變更都有可靠的落地機制。這四個工具共同構成的，不只是一個更穩定的研究平台，更是一套讓未來論文結論更可信的基礎設施。

研究工具和生產服務的界線，正在 RAG_Ultimate 上一點一點地消融。
