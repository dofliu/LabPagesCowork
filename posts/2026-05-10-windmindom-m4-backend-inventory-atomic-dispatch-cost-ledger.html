一、M4 的核心挑戰：庫存與成本的跨域一致性

離岸風場的備品管理，遠比陸上工廠複雜。一張申料單（Material Request）從提交到實際出庫，需要橫跨三個部門核准（技術員 → 主管 → 財務主管），且核准後才能觸發倉庫的庫存扣減。問題的關鍵在於：當申料單被核准並派工時，如何確保庫存扣減與成本記帳在同一個 transaction 中完成？ 任何部分失敗都可能導致「帳上料已派出但庫存未扣」或「庫存扣了但帳沒記」的資料不一致。

另一個更細膩的會計挑戰是：料件單價可能在申料核准後發生變動。例如，派工時備品單價為 NT$450，但一個月後供應商漲價到 NT$600；這時工單完工後的「確認成本」應該用哪個價格計算？若用當下價格（$600），帳目無法對應到當初的估算依據；若用估算時的單價（$450），才符合財務「成本確認以承諾時點為準」的原則。

💡 離岸風場成本管控的特殊情境

在離岸風場運維合約中，備品成本往往是費用糾紛的主要來源。運維廠商向業主（風場開發商）請款時，必須提交每張工單的完整成本憑證：包含備品規格、申料日期、核准過程、實際領料數量，以及成本計算依據。若系統無法提供「申料核准當下的鎖定單價」，財務對帳時就難以自証清白。windMindOM 的 locked_unit_cost 欄位正是為了解決這個實務痛點而設計。

二、五個 Issue 的推進順序

M4 backend 的推進嚴格遵循由內而外（Domain → Repository → API）的 DDD 開發順序，確保每一層建立在穩固的下層之上：

三、Atomic Dispatch：原子雙寫的設計

A2 是整個 M4 的技術核心。當申料單通過三階審批後，系統必須同時完成兩件事：（1）扣減倉庫庫存、（2）在成本台帳寫入 estimated 記帳分錄。這兩個操作橫跨 workflow 與 cost 兩個模組——在 DDD 語境中屬於不同的 Bounded Context——如何確保它們要麼全成功，要麼全失敗？

windMindOM 的解法是讓 cost_ledger 與 workflow repository 共用同一個 SQLAlchemy Session Base，透過傳遞 session 物件實現同一 transaction 內的雙寫，而非使用分散式 saga 或補償機制。這是一個務實的「單體優先（Monolith-first）」設計決策：在 MVP 階段，過早引入 saga/outbox pattern 的複雜度遠高於其收益；當未來真正需要拆服務時，清晰的邊界設計讓拆分變得可行。

🔧 Atomic Dispatch 的完整生命週期

# 建料件（dispatch 時鎖定此單價）
POST /api/workflow/inventory → unit_cost=450

# 建工單 → 申料 → 三階核准
POST /api/workflow/work-orders → OPEN → IN_PROGRESS
POST /api/workflow/material-requests → linked to WO
POST .../submit → POST .../approve (employee / leader / treasury)

# 審批完成：原子派工觸發
→ stock: 10 → 8 # 庫存扣減
→ ledger: estimated, amount=900, locked_unit_cost=450

# 供應商漲價（不影響已鎖定的帳目）
PATCH /api/workflow/inventory/{id} → unit_cost=600

# 收料 → 工單完工
POST .../receive → actual_qty=2
POST /api/workflow/work-orders/{id}/finish

# A5 完工 hook：用 locked_unit_cost 確認帳目
→ ledger: confirmed, amount=2×450=900 # 不是 2×600=1200 ✓
→ confirmed_at = 完工時間戳

四、Code Review 的三項 Must-Fix

backend 全部 issue 完成後，以 code-reviewer subagent 進行獨立審查，找到三項需立即修正的問題，這些修正將系統從「功能可用」提升至「生產品質」：

Fix 1 — 會計正確性：locked_unit_cost Schema Migration

原始設計中，cost ledger 的 confirmed amount 是在完工時重新讀取料件的 unit_cost 計算，這意味著若單價在派工後被修改，確認金額就會與估算金額不符——直接違反財務記帳原則。修正方案是在 A2 原子派工時，同步將當下的 unit_cost 快照寫入 ledger 的 locked_unit_cost 欄位，A5 完工確認時使用此快照而非即時單價。這個修正增加了 schema migration 和 6 個確認測試。

Fix 2 — Multi-Farm Safe：_finish_hook_db_path 改 dict

工單完工的 hook 設定（用於測試覆寫 DB 路徑）原本是一個全局字串變數，當同一程序內同時操作多個 farm 的工單時（例如 staging 環境的並發測試），不同 farm 的 hook 設定會互相覆蓋，導致非確定性測試失敗。修正為以 farm_id 為 key 的 dict，每個 farm 獨立持有自己的 hook 設定，消除 singleton 競爭條件。

Fix 3 — 精確 Status Code Mapping：InvalidTransition.reason 屬性

狀態機的無效轉換例外原本透過字串匹配來決定回傳 HTTP 400 還是 409，這種 fragile pattern 在例外訊息措辭略有變動時就可能誤判 status code。修正為在 InvalidTransition exception class 加上 reason 屬性（bad_request / conflict），讓 API router 直接讀取 reason 決定 status code，測試也更易理解與維護。

五、設計哲學：「事實帳本不可直接修改」

A4 庫存 API 的設計決策中，有一條明確的規定：不暴露直接修改 cost_ledger 的 POST/PATCH endpoint。這不是技術限制，而是領域規則：成本台帳是業務事實（business fact）的記錄，所有寫入都必須透過業務流程的 atomic transaction 觸發——申料核准觸發 estimated entry，工單完工觸發 confirmed entry，未來的退料申請會觸發 ledger 沖銷（F1 follow-up issue）。

這個原則確保了台帳的每一筆記錄都能溯源到具體的業務事件。在未來的月報生成（A8 reporting）和客戶請款憑證中，審計員可以從任何一筆 ledger entry 向上追溯到對應的工單、申料單、審批記錄，形成完整的業務證明鏈。這正是 Domain-Driven Design 中 Aggregate Root 概念的實務體現：台帳的 aggregate root 只能透過定義好的 domain service 方法修改，外部不得繞過。

💡 為什麼 Lazy Import 解循環依賴？

A5 的實作過程中發現一個循環依賴：material_request_repository 需要引用 cost_ledger_repository 來寫入台帳，而 cost_ledger_repository 又需要引用 workflow.repository 的 Base。這種跨 Bounded Context 的相互引用在 Python 的模組系統中會產生 ImportError。解決方案是在需要跨域引用的函數內使用 lazy import（將 import 語句移至函數體內而非模組頂層），讓 Python interpreter 在實際呼叫時才解析依賴，而非在模組載入階段。這是 Python 項目中處理有意義的跨域依賴的標準技法，比強行重構兩個模組的邊界更為務實。
