RAG_Ultimate 科學可重現性衝刺：Mulberry32 PRNG 為風電 RAG Benchmark 論文奠基

2026-04-17 • 劉瑞弘教授團隊 • RAG_Ultimate / Reproducibility / Mulberry32 / Seeded Random / Benchmark

5
Phase 3 風電 RAG Benchmark 論文：5 個關鍵實驗倒數準備

一篇好的 Benchmark 論文，和一篇能被期刊退稿的 Benchmark 論文，差別往往不在演算法本身，而在「這些數字能不能被重跑一次」。今天凌晨，RAG_Ultimate 團隊一口氣合併了 Issue #56 ——一次修復 17 處 Math.random() 缺少亂數種子的問題。從表面看這只是一次例行的 quality sprint，但對即將啟動的 Phase 3 風電 RAG Benchmark 論文 而言，這一步卻是決定性的：它讓即將產出的實驗數據具備可重現性（reproducibility），也讓 4 個品牌、405 題評測集、17 種 RAG 模式、5 次重複執行的複雜實驗矩陣，有了值得被同儕檢視的科學基礎。

本文從 DOF Lab 的角度，拆解這次修復背後的技術選擇，為什麼「可重現性」在 2026 年的 LLM 研究社群格外重要，以及接下來 5 個實驗的跑法。

一、為什麼 Math.random() 會讓一篇 benchmark 論文失去說服力

在 JavaScript / TypeScript 前端環境中，Math.random() 是一個不可設定種子（unseedable）的 PRNG。每次呼叫都會產生不可預測的浮點數——對遊戲、動畫特效來說很好，但對科學實驗是災難性的。想像論文中寫：「我們在 104 題 Z72 評測集上跑了 12 種 RAG 模式，每種跑 5 次，取 MRR 平均值」——如果任何一個步驟使用了 Math.random()，例如：

  • AutoResearch 引擎：用隨機搜索挑選超參數組合
  • Adversarial Test Service：隨機擾動 query 測試穩健性
  • Elo Ranking Service：Pairwise 比較時隨機採樣對戰組合
  • Statistics Utils：Bootstrap 信賴區間抽樣

那麼即使同一組人、同一個 commit、同一份資料，跑出來的數字也會每次不一樣。這在工程上可能被接受，但對一篇要投到 IEEE Access 或 Applied Energy 等學術期刊的論文，審稿人只要問一句「你的實驗能否被我在自己的機器重跑出相同結果？」——答案如果是「不能」，那 95% confidence interval 的統計檢驗就失去根基。

⚡ Issue #56 的 17 個修復點分成兩類：
  • 高風險（影響實驗結果，14 處）：全部改用 Mulberry32 seeded PRNG，透過外部傳入 seed 參數控制亂數序列
  • 低風險（僅做 ID 生成，3 處）：改用 crypto.randomUUID()——本質上是要全域唯一，不需要可重現

二、為什麼選 Mulberry32，而不是直接用 seedrandom 套件

Mulberry32 是由 Tommy Ettinger 提出的一個極簡 32-bit 狀態 PRNG，只需要 6 行程式碼就能在 JavaScript 中實作。它的週期約為 2³² ≈ 42 億，對單次實驗跑數萬次取樣已綽綽有餘。RAG_Ultimate 的實作在 utils/seededRandom.ts：

export function createSeededRandom(seed: number): () => number { let s = seed | 0; return () => { s = (s + 0x6D2B79F5) | 0; let t = Math.imul(s ^ (s >>> 15), 1 | s); t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t; return ((t ^ (t >>> 14)) >>> 0) / 4294967296; }; }

比起引入 seedrandom 這種第三方套件，自寫 Mulberry32 有三個優點：沒有外部相依、可讀、可審計。對於一篇 benchmark 論文的「實驗方法」章節，審稿人可以直接在內文附上這 7 行程式碼，並保證任何讀者可以在任何語言中重新實作出完全相同的亂數序列。這種「論文自給自足」的可重現性，是過去幾年 NeurIPS、ICML 陸續推動「Reproducibility Checklist」的核心訴求。

更重要的是，整個 RAG_Ultimate 的實驗流程現在都接受 seed 參數——從使用者介面的 AutoResearch 實驗配置，到後端 bootstrap 抽樣，到 Elo pairwise 比較，全部可以用同一顆 seed 完整重現。一鍵「以 seed=42 重跑實驗 1」將變成日常操作。

在 Mulberry32 修復之前，實驗 1 的 5 次重複執行會因為 AutoResearch 引擎的隨機性導致變異數不可控：研究者無法分辨「這個變異來自 LLM stochasticity」還是「來自我們程式碼中未受控的隨機取樣」。修復之後，前者（模型本身的非確定性）仍保留——這是研究要觀察的本質；後者（工程端的雜訊）則被徹底消除。

四、2026 年 LLM 社群對可重現性的共識正在形成

這個方向並非單一實驗室的偏好。2026 年上半年，LLM 社群對「可重現性危機」的討論急劇升溫。ACL 2025 最佳論文之一就是檢討大型語言模型評測指標的可重現性問題，指出在 GPT-4 級別模型上，即使 temperature=0，不同 batch 大小與不同 tokenizer 實作也可能導致同一題答案不一致。LangChain 官方文件亦在 2026 Q1 新增「Reproducible Evaluations」章節，建議任何 RAG benchmark 都應記錄：seed 值、LLM 版本號、prompt 版本號、資料快照時間戳。

🔬 RAG_Ultimate 的實驗可重現性契約（Reproducibility Contract）：
  • Seed 固定：所有工程端隨機皆受 Mulberry32 控制，預設 seed=42
  • 模型版本釘選：Gemini 指定 model=gemini-2.5-flash-exp，Ollama 指定 bge-m3:latest digest
  • 資料快照：Qdrant 集合以時間戳命名，實驗結束自動備份為 JSON
  • Prompt 版本：所有 prompt 存於版本控管（git），實驗報告附 commit hash
  • 實驗日誌：每次 run 自動輸出 run_id、seed、時間戳、所有超參數至 audit log

五、同日未來：品質 Sprint 的下兩顆 issue

Issue #56 關閉的同時，auto-detected 機制立刻開出兩顆新的品質 issue，都是為了讓論文提交前的程式碼不留技術債：

  • Issue #58 — 後端 API URL 不一致：3 處重複定義 + 環境變數名稱衝突，會在 Docker Compose 部署時造成混淆
  • Issue #59 — 前端 12 處 as any 型別轉換：降低 TypeScript 安全性，特別是在 evaluation history 與 experiment tracker 這類核心模組

這兩顆 issue 不會阻擋實驗 1 啟動，但會被安排在實驗 1、2 跑的空檔處理掉，確保 v9.0 的程式碼庫在論文送審時能以「乾淨的工程實踐」作為補充材料被審稿人直接檢視。
