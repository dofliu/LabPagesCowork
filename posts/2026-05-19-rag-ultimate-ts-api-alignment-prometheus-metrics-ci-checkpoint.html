前後端型別契約收口：TypeScript ↔ Pydantic 對齊與 Prometheus 可觀測性端點

2026-05-19 • 劉瑞弘教授團隊 • RAG / TypeScript / Observability / Research Engineering

RAG_Ultimate TypeScript Prometheus CI Checkpoint Research Engineering

研究型軟體系統最常見的一種技術債，不是演算法錯誤，也不是架構選擇失當，而是一個安靜卻持續擴大的裂縫：前端以為 API 回傳的型別是 A，但後端實際回傳的是 B。這個差距在系統初期幾乎不可見，但隨著功能增加、介面演進，它會逐漸化為一連串神秘的 runtime 異常與「不知道為什麼」的資料呈現問題。

RAG_Ultimate 今日完成兩個相互補強的 PR：PR #110 專注於消滅這個「型別裂縫」，讓前端 backendApiService.ts 的 TypeScript 型別宣告與後端 schemas.py 的 Pydantic 定義完全對齊；PR #109 則新增 GET /api/v1/metrics 端點，以 Prometheus 格式暴露系統運行指標，為平台從「研究工具」邁向「可觀測生產服務」打下基礎。兩者合計，加上今日執行的 CI checkpoint #5，形成了本週例行品質維護週期中技術深度最高的一日。

C1：消滅型別裂縫——backendApiService 完整型別對齊

全棧 TypeScript 專案中，一個常見的痛點是前端服務層（service layer）隨著需求快速疊加，型別宣告逐漸「漂移」——有的欄位被標記為 any，有的 response shape 只有部分欄位宣告，有的則直接用 interface 猜測後端回傳格式。問題不在開發者不夠嚴謹，而在於全棧型別同步本身就需要耗費額外工夫，在快速研究迭代中常被暫緩處理。

PR #110（backlog C1，issue #8-a）做的事情看起來簡單：逐欄比對 schemas.py 中所有 Pydantic model 的欄位定義，然後在 backendApiService.ts 補上對應的 TypeScript interface。但這個「簡單」的工作需要跨越兩個型別系統的語義差距：

📌 Python Pydantic ↔ TypeScript 型別對應原則

Pydantic (Python)TypeScript 對應常見漂移風險
Optional[str] string | null | undefined 僅寫 string，None 傳回導致 runtime 崩潰
List[Dict[str, Any]] Record<string, unknown>[] 寫成 any[]，喪失後續重構的型別安全保護
Literal["cosine", "dot", "euclidean"] "cosine" | "dot" | "euclidean" 寫成 string，非法值在送出請求前無法被攔截
float（ge=0, le=1） number（搭配 JSDoc 邊界備註） TypeScript 無法表達數值範圍，需靠文件或 Zod 補強

這個對齊工作的意義不只是「讓 tsc 編譯通過」。更重要的是，它讓前端元件在呼叫 API 時能夠享有完整的 IDE 自動補全與靜態型別檢查——任何嘗試存取不存在欄位或傳入型別不符參數的操作，都會在開發期被 TypeScript compiler 攔下，而不是等到用戶在 UI 上操作時才以神秘的 undefined is not a function 浮現。

🔍 禁區外折衷策略
PR #110 刻意僅觸碰 backendApiService.ts（前端服務層），而非深入 core services/ 禁區。這遵循 ROUTINE.md 的「禁區外部署」原則：每個例行 PR 的影響範圍清晰可控，不引入意外的核心邏輯改動。

E1：讓系統「說話」——Prometheus Metrics Endpoint

在 RAG 研究平台的發展脈絡中，「可觀測性（Observability）」常被視為「等平台穩定後再說」的事。但這個決定有一個潛在代價：當你終於需要了解系統行為時——「為什麼這個查詢的延遲突然變高？」「Qdrant 的連線佇列是否飽和？」「哪種 RAG 模式佔用最多 embedding API 配額？」——你沒有任何資料可以回答。

PR #109（backlog E1，issue #10-a）新增了 GET /api/v1/metrics 端點，以 Prometheus 文字格式（text/plain; version=0.0.4）暴露系統指標。選擇 Prometheus 格式而非自定義 JSON 有幾個具體理由：

📊 /api/v1/metrics 端點設計

# HELP rag_requests_total Total number of RAG requests processed # TYPE rag_requests_total counter rag_requests_total{mode="hybrid",status="success"} 1024 rag_requests_total{mode="dense",status="success"} 876 rag_requests_total{mode="sparse",status="error"} 12 # HELP rag_retrieval_latency_seconds RAG retrieval latency in seconds # TYPE rag_retrieval_latency_seconds histogram rag_retrieval_latency_seconds_bucket{le="0.1"} 234 rag_retrieval_latency_seconds_bucket{le="0.5"} 891 rag_retrieval_latency_seconds_bucket{le="+Inf"} 912 # HELP qdrant_collections_total Number of Qdrant collections # TYPE qdrant_collections_total gauge qdrant_collections_total 7

（示意格式，實際欄位依 schemas.py MetricsResponse 定義）

Prometheus 格式的最大優勢在於生態系統的相容性：任何支援 Prometheus scrape 的監控工具——Grafana、Alertmanager、Datadog、甚至 VS Code 的 Prometheus 擴充套件——都可以直接接入，無需額外的資料轉換層。對於一個研究平台來說，這意味著未來可以以極低的成本接上視覺化儀表板，讓研究者在進行 RAG benchmark 實驗時即時看到系統負載與延遲分布。

💡 可觀測性對 AI 研究平台的特殊意義

一般 web 服務的 metrics 主要關心 p99 延遲與錯誤率。但 RAG 研究平台還需要追蹤一類特殊的指標：檢索品質的代理指標——例如不同 RAG 模式被呼叫的頻率比例（間接反映研究者偏好）、score threshold 過濾掉的文件佔比（反映知識庫品質），以及 embedding 模型切換的頻率（反映實驗密度）。這些指標不存在於傳統 web metrics 的詞彙表中，但對於理解「研究平台在被如何使用」至關重要。PR #109 建立的 metrics 端點，是收集這類研究特定指標的基礎設施起點。

CI Checkpoint #5：1730 測試全通的意義

依照 RAG_Ultimate 的例行節奏規範，每累積 5 個合併 PR 需執行一次 CI checkpoint：在不發出任何 PR 的情況下，對本地程式碼庫進行完整的「乾跑」，確認所有積累的改動沒有引入隱性的交叉破壞。今日 checkpoint #5 的結果如下：

檢查項目結果說明
npx tsc --noEmit ✅ 0 errors TypeScript 全棧型別對齊完整，無任何型別逃逸
jest --coverage ✅ 1730/1730 86/88 test suites 通過（2 件 legacy pre-existing fail，非本週引入）

1730 個 Jest 測試全數通過，這個數字背後代表的是過去數週「清潔工周期（Code Health）」、「Schema 強化周期」與今日「型別對齊＋可觀測性」三個不同方向改動的疊加，沒有引入任何測試回歸。對於一個仍在快速迭代的研究平台而言，這是一個有意義的穩定性信號。

⚠️ 例行程序的「4 連 Skip」現象

值得記錄的是，PR #110 在等待 merge 期間，例行品質電池連續 4 輪（r31–r34）觸發後全部跳過執行——這追平了歷史次高紀錄（r25–r28 的 4 連 skip）。系統的行為完全正確：每輪 log 忠實記錄 PR 狀態，不擅自並行開新 PR。這種「高能見度、不越界」的設計原則確保了 AI 輔助的例行維護程序不會失控地堆疊未 review 的改動。然而，這也提示了一個流程優化空間：PR merge 的確認迴路若能縮短，整個 Cycle 的吞吐量就能顯著提升。

從「研究工具」到「可觀測服務」的路徑

回顧今日的兩個 PR，它們代表著 RAG_Ultimate 發展方向的一個微妙轉折：過去的工作重心在於「能跑起來、能出結果」（RAG 模式實作、評測框架、UI 重構）；而今日的工作——型別契約的嚴格化、Prometheus metrics 的接入——更接近於「讓這個系統變得可信任、可維護、可監控」。

這個轉變在研究工程學中有個常見的名稱：從 prototype fidelity（原型忠實度）到 production readiness（生產就緒度）。對於一個以論文產出與教學演示為目標的大學研究平台來說，「生產就緒度」不代表要部署到真實用戶，而是代表：平台的行為是可解釋的、問題是可追蹤的、改動是可驗證的。今日的工作，正是在這條路上邁出了兩步。
