多代理任務狀態汙染的系統性根治：WindAILab Task Session 架構解剖

2026-04-30 • 劉瑞弘教授團隊 • WindAILab / 多代理系統 / WebSocket / 系統架構

WindAILab 多代理系統 WebSocket Task Session

當你建構一個能執行完整風力發電診斷管線的多代理系統——從 SCADA 資料清洗、特徵工程、ML 故障分類，到最後的自然語言報告生成——你所面對的挑戰不只是 AI 精度，而是長時程、多步驟工作流程中的狀態管理。WindAILab Phase 14b 的核心工作就是系統性地解決一個問題：為什麼執行第二次診斷任務時，舊任務的圖表和步驟紀錄會「滲透」進來？

這個看似 UI Bug 的問題，實際上反映了多代理系統設計中的一個普遍陷阱：全域狀態（Global State）與任務邊界（Task Boundary）的混淆。此篇文章將完整解剖 WindAILab 找到的 6 個根因，以及用 Task Session 架構一次性根治的設計思路。

一、症狀清單：那些奇怪的現象

在 Phase 14b 之前，WindAILab 的使用者會遇到一系列難以重現但反覆出現的問題：

症狀 根因
單次診斷產生 3 筆任務紀錄 前端 Simulation + Backend 雙重執行 + debounce 失效
新任務串接舊任務的工作步驟 WorkflowProgress 讀取全域 workLogs，無任務邊界
圖表跨任務累積（舊圖表殘留） analysisResults 全域陣列，從未在任務開始時清空
Refresh 後所有圖表消失 broadcast_analysis_result 只廣播，從未寫入 SQLite
進度條顯示 8700% / 帶小數點 前後端雙重百分比乘法 + 未 Math.round()
工作日誌 Refresh 後消失 WebSocket 廣播不儲存，無後端 log buffer

這些問題單獨看都像是小 Bug，但放在一起，它們揭示了一個設計層面的問題：系統的狀態管理沒有「任務」這個概念作為基本單位。代理的輸出、UI 的圖表、進度的計算，全部共用同一塊全域記憶體空間。

二、Task Session 架構：讓每次診斷都是全新的開始

Phase 14b 的核心解法是引入 Task Session（任務會話）的概念，讓後端 WebSocket 事件成為任務生命週期的唯一真實來源（Single Source of Truth）。架構的關鍵改變如下：

🔧 Task Session 設計原則

  • task_started 事件：後端廣播時，前端自動清除所有舊的 analysisResults、workLogs、步驟紀錄，確保每次任務從零開始。
  • task_completed 事件：任務結束的唯一信號，前端根據此事件更新任務清單，不再依賴前端 simulation。
  • Log Buffer 機制：WebSocketManager 內建 per-task log buffer，Refresh 後前端可重新取得完整工作日誌。
  • 結果同步寫入 SQLite：broadcast_analysis_result 同時呼叫 db.save_analysis_result()，圖表持久化不依賴記憶體。
  • WorkflowProgress 切片：只傳入「當前任務啟動後」的 logs（slice from task_started），隔離任務邊界。

這個設計的精妙之處在於，它把原本分散在前端各元件的狀態假設，集中到後端 WebSocket 事件流。前端不再需要猜測「現在是哪一個任務在跑」，只需監聽 task_started 事件並重置本地狀態即可。

三、更深的啟示：多代理系統的「任務邊界」問題

WindAILab 的這個工程經驗，對所有構建多代理 AI 系統的開發者都有參考價值。當系統中有 25 個以上的 AI 代理協同運作，而每個代理都可能非同步地產出中間結果（分析圖表、診斷報告、進度更新），如何讓 UI 端清楚知道哪些輸出屬於哪個任務，是一個容易被忽視但影響深遠的問題。

⚠️ 常見的多代理狀態陷阱

  • Event 無任務 ID：後端廣播事件時未攜帶 task_id，前端無法區分不同任務的輸出。
  • 前端 Optimistic Update：前端提前模擬任務進度（simulation mode），與後端實際狀態產生競爭，導致重複計算。
  • 非同步清除不一致：圖表清除發生在任務啟動前，但代理輸出可能延遲到達，導致「新任務啟動 → 舊結果才到 → 顯示混亂」的 race condition。
  • 持久化與廣播解耦：只廣播不儲存，導致 WebSocket 斷線或 Refresh 後狀態遺失。

Task Session 架構的解法是：以後端事件驅動為中心，前端成為純粹的「狀態訂閱者」。每個任務有清晰的開始（task_started）與結束（task_completed），所有中間輸出都帶著 task_id 標籤，UI 只渲染當前活躍 session 的資料。這在設計上接近 Redux 的 Action-based 狀態機，但以 WebSocket 事件流作為 Action Source。

四、Phase 14 的系統性演進意義

WindAILab 的 Phase 14 不只是功能累加，而是一次系統從「研究原型」邁向「生產就緒」的架構升級。Phase 14a 帶來了 WindGuard AI 的 LLM 推理能力，Phase 14b 解決了長時程任務的狀態管理，Phase 14c 則完成了報告的持久化與 PDF 輸出。三個子階段合在一起，構成了一個可以放心交給非技術使用者操作的診斷服務平台。

從研究角度來看，WindAILab 現在的狀態意義在於：我們不只能跑出 F1=1.0 的故障診斷精度，還能在一個工程穩健的系統中持續累積診斷資料、生成可引用的報告——這正是從論文成果走向實際風場應用的關鍵一步。

🔬 DOF Lab 研究室行動項目

  • Phase 14d 預計推進：外部資料連接器（外部 SCADA API 直接注入診斷管線），實現零人工介入的自動巡檢排程。
  • Task Session 架構將作為標準模式，應用於 InduSpect 工業巡檢系統的多代理設計，避免重蹈相同設計坑。
  • WebSocket 事件設計規範化：統一所有代理廣播格式，確保 task_id、timestamp、agent_id 三欄必填，建立代理輸出的可追溯性。
  • 進行 bug 模式整理，形成「多代理 UI 狀態管理反模式」文件，納入研究室開發規範。
