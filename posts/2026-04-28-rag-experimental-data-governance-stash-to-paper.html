RAG 實驗資料治理：從 Git Stash 到論文主表的可重現性之路

2026-04-28 • 劉瑞弘教授團隊 • RAG_Ultimate / 實驗資料治理 / 版本控制 / 研究方法論

RAG 實驗 資料治理 版本控制 研究方法論

在 AI 研究中，有一種困境幾乎每個做嚴謹評測的團隊都曾遇到：實驗跑出來了，結果也有了，但資料卻散落在本機磁碟、臨時目錄，甚至只存在 git stash 裡。RAG_Ultimate 研究團隊正面對這個挑戰——Tier-2 audit 的完整文件仍在 stash@{0}，尚未正式納入版本控制；與此同時，r06/r07 補跑任務也在等待排程，以確保論文主表的每個檢索模式至少擁有 4 次完整實驗輪次（≥4 complete runs）。

這不只是一個「整理資料夾」的問題，而是牽涉到整個 AI 研究生命週期的資料治理（Data Governance）設計。2026 年，隨著 AI 研究的規模不斷擴大，實驗資料的版本追蹤與透明化管理已成為學術界和工業界共同重視的核心課題。

一、「研究債」的隱形代價

軟體開發中有「技術債（Technical Debt）」的概念，AI 研究中同樣存在類似的「研究債（Research Debt）」：當實驗結果的整理速度跟不上產出速度時，這些未被正式版本化的資料就成為隱形負擔。常見的表現形式包括：

  • 實驗結果只存在本機，未 commit 進 Git repository
  • 使用 git stash 暫存大量 checkpoint 與 audit 文件，無明確的解封策略
  • 因網路中斷、環境不穩定等外部因素導致部分 run 不完整，但記錄不夠清楚
  • 補跑任務排程不明確，導致論文主表的信賴區間無法收斂

在 RAG_Ultimate 的 Tier-2 benchmark 情境中，這些問題特別敏感：17 種檢索模式的比較若有任何一輪資料遺漏，就可能讓統計結論的可信度大幅下滑。正因如此，「從 stash 到論文主表」的資料治理路徑，值得系統性地思考。

⚠️ 研究債的三大警訊

當以下情況出現時，代表實驗資料治理需要立即介入：

  • Stash 累積過多：git stash list 出現多個未命名 stash，且已超過 1 週未解封
  • Run 完整性不明：無法快速回答「目前有幾個 mode 達到 ≥4 完整 run」
  • 版本與結果解耦：看到一份 CSV 結果檔，但無法確定對應哪個 commit、哪個參數配置

二、2026 年 AI 實驗資料治理的最佳實踐

根據 2026 年業界與學術界的共識，AI 實驗的資料治理已不再只是「記得 commit」這麼簡單，而是需要將治理邏輯嵌入整個研究工作流程。以下是幾個核心原則：

治理面向 舊有做法（研究債高風險） 2026 建議做法
實驗結果存放 本機目錄、臨時 stash 命名分支 + DVC / MLflow 追蹤
Run 完整性追蹤 人工記錄 / 靠記憶 自動化 watchdog（如 tier2_watchdog.ps1）
Audit 文件納版 每次完成後手動決定 PR 模板要求 audit log 作為 checklist
補跑排程 依賴研究者記憶觸發 Issue 追蹤 + 自動化 CI 排程提醒
版本與參數綁定 結果檔與設定檔分離 每次 run 攜帶 config hash 作為檔名前綴

💡 核心觀念：把 Governance 當成 Engineering 問題

2026 年最重要的轉變在於：AI 資料治理不再是研究後期的「清理工作」，而是在第一行程式碼寫下之前就要設計好的工程架構。就像 CI/CD pipeline 在程式碼 push 前就自動跑測試，實驗 governance pipeline 應該在 run 啟動前就確認追蹤機制已就位。

三、RAG_Ultimate 的具體行動路徑

面對目前的情況，DOF Lab 的 RAG_Ultimate 研究團隊規劃了以下的資料治理行動路徑，將 stash 中的 Tier-2 audit 文件轉化為論文可引用的正式實驗記錄：

  1. 解封 stash@{0}：將 Tier-2 audit 文件從 stash 移出，以命名分支（如 feat/tier2-audit-commit）納入版本控制，並附上完整的 commit message 說明資料來源、產生時間與使用的參數配置。
  2. 補跑 r06/r07：針對目前不足 4 complete runs 的 RAG mode，在穩定網路環境下啟動補跑任務。每次 run 結束後立即 commit 結果，不允許再次進入 stash 狀態。
  3. 更新 Tier-2 主表：整合所有 ≥4 complete run 的模式，計算各模式的平均分數與 95% 信賴區間（目標 ±0.03），產出論文主表（Table 1）的最終版本。
  4. 建立 audit trail：為每個 RAG mode 的最終結果建立 audit log，記錄「使用哪個 commit 的程式碼、哪批資料、哪個 seed」，確保任何審稿人都能重現結果。
  5. PR 合規檢查：在論文主表 PR 中加入 checklist，確認所有 17 個模式皆達到完整性要求，CI 通過，才允許 merge 進 main 分支。

🚀 DOF Lab 研究室行動項目

  • 近期優先：決定 stash@{0} 的 Tier-2 audit 文件處理策略，建議以命名分支方式納版
  • 本週排程：r06/r07 補跑，目標讓所有 17 個 RAG mode 達到 ≥4 complete runs
  • 工具化方向：評估為 RAG_Ultimate 建立一個 run-tracker dashboard，即時顯示每個 mode 的完整度狀態
  • 文化建立：推動「實驗完成即 commit」的研究習慣，避免 stash 成為實驗結果的黑洞

結語

「從 stash 到論文主表」看似只是一個版本控制的小問題，但背後反映的是整個 AI 研究過程對資料治理的重視程度。在競爭激烈的 AI 研究環境中，可重現性（reproducibility）與透明度（transparency）不只是學術倫理要求，更是研究成果能否被引用、被信任的關鍵基礎。

RAG_Ultimate 即將進入論文收尾階段，17 種檢索模式的 benchmark 結果將成為風電領域 RAG 研究的重要參考。嚴謹的資料治理，將確保這份成果經得起時間與同儕的考驗。
