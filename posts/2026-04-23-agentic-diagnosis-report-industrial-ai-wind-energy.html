從分數到決策：工業 AI 診斷報告自動化的關鍵突破與 WindAILab Phase 14c 前瞻

2026-04-23 • 劉瑞弘教授團隊 • 工業 AI / 診斷報告 / Agentic AI / WindAILab

診斷報告自動化 WindAILab Phase 14c Agentic AI

工業 AI 的研究社群，長期以追求「更高精度的模型」為第一優先。F1-score 要達到 0.99、R² 要突破 0.997——這些數字固然重要，但一個殘酷的現實是：即便模型在測試集上表現完美，如果它的輸出結果無法被現場工程師理解與採用，它的實際價值等於零。

DOF Lab 在 WindAILab 平台的最新階段（Phase 14c），正是要解決這道「最後一哩路」問題：如何讓高性能 AI 模型的診斷結論，自動轉化為工程師可以直接採取行動的結構化報告？ 這篇文章將梳理工業診斷報告自動化的技術挑戰、業界最新進展，以及我們即將推進的設計方向。

一、工業 AI 的「可信度落差」問題

根據 McKinsey 2025 年的工業 AI 報告，超過 60% 的工廠試點 AI 專案最終停留在 Proof-of-Concept（PoC）階段，無法進入生產部署。其中最常見的原因，並不是模型精度不夠，而是工程師不信任 AI 的輸出——他們看到的只是一個異常分數（如 0.87）或一個分類標籤（如 "Bearing Fault"），但不知道：為什麼 AI 這樣判斷？依據是哪些感測器資料？建議的維護動作是什麼？

這個問題在風力發電領域尤為突出。一座海上風場的單台風機停機損失，每天可能超過數萬美元。若運維工程師基於 AI 誤報而緊急停機、派遣海上人員作業，成本同樣高昂。AI 不僅要「對」，更要能「說清楚」。

⚠ 挑戰一：黑盒輸出

深度學習模型（LSTM、Transformer）給出分數，但缺乏人類可讀的因果解釋。工程師難以驗證診斷邏輯。

⚠ 挑戰二：跨系統資料孤島

SCADA 資料、天氣資料、歷史維護紀錄、零件壽命資料散落在不同系統，診斷報告需整合多源資料才有意義。

⚠ 挑戰三：報告格式不統一

不同廠商、不同機型的報告格式各異，難以建立可複用的自動化生成流程，也無法積累跨風場的知識庫。

⚠ 挑戰四：動作建議缺失

多數系統僅輸出「異常警告」，缺乏明確的維護建議、優先級排序與預計工時，讓排程決策仍依賴人工判斷。

二、Agentic AI 如何重新定義工業診斷報告

2025 至 2026 年間，Agentic AI 的興起為工業診斷報告帶來了根本性的改變思路。傳統做法是「模型輸出分數 → 人工撰寫報告」，Agentic AI 的做法則是：一群具備不同專業的 AI Agent 協同運作，自動從感測器資料到結構化報告完成整個流程。

🤖 Agentic 診斷報告的核心能力

多 Agent 協作：數據讀取 Agent 負責拉取 SCADA 資料，分析 Agent 執行模型推理，解釋 Agent 生成人類可讀的因果說明，報告 Agent 統整輸出為結構化文件。每個角色明確，輸出可追溯。

RAG 知識庫支援：透過 Retrieval-Augmented Generation，診斷報告可自動引用歷史案例、製造商手冊、法規標準，讓建議動作有文獻依據而非憑空生成。

結構化輸出協議：採用 JSON Schema 或 Structured Output 技術，確保報告欄位（故障部位、嚴重程度、建議動作、預計工時、置信度）完整且機器可讀，可直接匯入 CMMS 維護管理系統。

業界已有先行者在這條路上取得重要進展。Siemens Energy 的 AI 運維平台在 2025 年整合了 LLM 驅動的報告生成模組，將診斷到報告的平均耗時從 4 小時縮短至 15 分鐘。GE Vernova 的 APM（Asset Performance Management）系統則透過 Agent 架構，讓診斷報告自動觸發下游的備件採購工單。

三、WindAILab Phase 14c 的設計藍圖

WindAILab 在完成 Phase 14b（PatchTST 多模型對比框架、告警規則引擎）後，Phase 14c 的核心任務是打通「診斷結果」到「結構化報告」再到「資料連接器」的完整鏈路。

①

診斷結果聚合層

整合來自 NBM 異常分數、LSTM 預測偏差、PatchTST 趨勢警示三個模型的多維輸出，統一進入結構化 DiagnosisContext 物件，包含故障部位、時間戳、置信度分佈與觸發規則 ID。

②

LLM 報告生成 Agent

以 Google Gemini API 為後端，接收 DiagnosisContext 並透過 Structured Output 生成標準化診斷報告。報告包含：執行摘要（2-3 句）、根因分析（附 SCADA 圖表引用）、風險等級（Critical / Warning / Info）、建議動作清單（含優先級與預計工時）。

③

RAG 知識庫整合

在報告生成時自動檢索 ChromaDB 中的歷史案例與技術文件，讓每一條建議動作都能附上「類似案例參考」，提高工程師的信任度與報告的可追溯性。

④

資料連接器與外部整合

透過 FastAPI 提供 REST API 端點，讓報告可推送至 Slack 通知、Google Sheets 記錄、或未來的 CMMS 系統。MCP（Model Context Protocol）工具鏈作為連接器標準，確保各下游系統可靈活插拔。

📐 結構化診斷報告的欄位標準（Phase 14c 草案）

我們參考 IEC 61400-25（風力發電監控通信協議）與 ISO 13381-1（預測維護資訊規範），設計了以下核心報告欄位：

  • report_id：UUID 格式唯一識別碼
  • turbine_id：風機識別碼（對應 SCADA 資產編號）
  • diagnosis_time：ISO 8601 格式時間戳
  • fault_components：故障部位列表（如 gearbox、main_bearing）
  • severity：Critical / Warning / Info 三級制
  • confidence_score：0.0–1.0 模型置信度
  • root_cause_summary：LLM 生成的自然語言根因說明
  • recommended_actions：有序動作清單，含優先級與預計工時
  • evidence_refs：RAG 知識庫中的參考案例 ID 列表
  • model_votes：各模型（NBM / LSTM / PatchTST）的投票結果

四、為什麼「資料連接器」是報告自動化的基石

診斷報告的價值，不在於報告本身，而在於報告被「接住」之後觸發了什麼行動。如果報告只是儲存在伺服器上等人查閱，它與傳統的告警日誌並無本質差別。真正讓報告進入決策閉環，需要資料連接器（Data Connector）扮演橋梁角色。

Phase 14c 計畫實作三類連接器：首先是通知連接器——當嚴重性為 Critical 的報告產生時，自動推送 Slack 訊息或電子郵件給當值工程師；其次是記錄連接器——將報告結構化欄位寫入 Google Sheets 或資料庫，形成可分析的歷史診斷庫；第三類是工單連接器（未來規劃）——直接在 CMMS 系統中建立維護工單，消除人工轉填的摩擦。

值得一提的是，MCP（Model Context Protocol） 在這裡發揮了重要的架構角色。MCP 提供了一套標準化的工具呼叫協議，讓 AI Agent 可以用統一的介面調用不同的外部服務，而不需要為每個目標系統撰寫專屬的整合程式碼。WindAILab 的連接器層將優先以 MCP 工具包形式封裝，與 DOF Lab MCP 生態系統保持一致。

🚀 DOF Lab Phase 14c 研究行動項目

  • DiagnosisReport Schema 定版：完成欄位設計，輸出 JSON Schema 與 Pydantic Model
  • Gemini Structured Output 整合：實作 LLM 報告生成 Agent，驗證欄位完整率 ≥ 98%
  • RAG 知識庫對接：確認 ChromaDB 檢索到的案例能正確注入報告的 evidence_refs 欄位
  • Slack / Sheets 連接器實作：完成兩個基礎連接器，端對端測試觸發流程
  • 論文撰寫準備：以 Kelmarsh SCADA 資料集為實驗基準，準備 Case Study 章節素材

工業 AI 的未來，不只是更好的模型，而是更完整的感知-分析-溝通-行動閉環。WindAILab Phase 14c 正在補上這個閉環中最重要、也最常被忽略的一塊：讓 AI 不只能「診斷」，更能「說清楚」，最終讓工程師與系統能放心地基於 AI 輸出採取行動。
